# 記事4: 4つのAIに同じ設計判断を聞いたら、全員違うことを言った

## ターゲット読者

- AIを業務で使っているが、1つのAIだけに依存している人
- 設計判断や意思決定にAIを使いたいエンジニア/PM
- 非エンジニアでも「AIの使い分け」に興味がある人

## 主張

AIの意見は材料であって結論ではない。
1つのAIに聞くと「答え」に見える。4つに聞くと「選択肢」に見える。
複数AIの意見を集めると、人間の判断の質が上がる。

## 構成

### 1. 導入: 誤検知率85%の絶望（200字）

- テスト設計チェッカーexspecを作った。自分のプロジェクトではうまく動いた
- OSSのLaravel（10,790テスト）にぶつけたら、BLOCKの85%が誤検知だった
- 設計を根本から見直す必要がある。でもどう直すか分からない
- 4つのAIに同じ質問をぶつけてみた

### 2. 質問: MLか、パターン追加か（300字）

- T001（assertion-free）の誤検知が多すぎる
- 選択肢A: パターンを手動で追加し続ける。確実だが際限がない
- 選択肢B: MLで分類する。汎用的だが学習データと精度の問題
- 選択肢C: ユーザーに設定で逃がす（escape hatch）
- 4つのAIに同じプロンプトで聞いた

### 3. まず全員が一致したこと（300字）

分岐点より先に、合意点を出して記事の軸を安定させる:

**全員一致（3つ）:**
1. MLはT001には不要。パターンが有限で列挙可能だから
2. escape hatch（ユーザー設定）は必要
3. 現在の「assertの有無を数える」アプローチは不十分

4者とも「ML不要」は即答。これだけでも、1つのAIに聞くより確信度が上がった。

### 4. 4つの回答 -- 同じ問いに対する異なるアプローチ（1,200字）

各AIの回答を「今回の問いに対してこう振る舞った」という書き方で紹介。性格の断定は避ける。

**比較表:**

| AI | 要点1文 | 採用 | 理由 |
|----|--------|------|------|
| GPT | 「oracle-free」にリフレーム | 採用: 設計原則 | 問題の再定義で視野が変わった |
| Claude | root→modifier→terminal の分類体系 | 採用: 実装構造 | 構造化が実装に直結した |
| Gemini | import-based自動拡張 | 保留: 将来課題 | 設計は魅力的だが今は重い |
| Grok | escape hatch最優先 | 採用: 同時実装 | ユーザー視点の補完 |

**GPT: 問題をリフレーミングした**
- 「T001はassertion-freeではなくoracle-free。テストオラクルを検出すべきだ」
- この一言で問題の見え方が変わった
- assertの有無を数えるのではなく、テストが「何かを検証しているか」を見る

**Claude: 構造を整理した**
- パターンを分類: root型（expect/assert）、修飾子型（.not/.resolves）、終端型（メソッド/プロパティ）
- 「オラクルの形」という構造を提示。チェーンの深さ問題を指摘
- 「MLは早すぎる」と明確に反対

**Gemini: 将来の拡張を提案した**
- import文を解析して、vitest/chaiをimportしているなら拡張パターンを自動有効化する提案
- 「ユーザーが設定しなくても正しく動く」方向
- 今回は保留したが、ロードマップの判断材料になった

**Grok: ユーザー体験を代弁した**
- `custom_patterns` escape hatchを最優先にすべき
- 「どんなにパターンを追加しても、ユーザー固有のhelperは認識できない。だから逃げ道が先」
- 技術的な正しさより「使えるか」

### 5. 人間が決めたこと（400字）

- GPTの「oracle-free」+ Claudeの分類体系 + Grokのescape hatchを組み合わせた
- Geminiの提案は将来課題として保留

**結果（dogfooding-results.mdより）:**
- Laravel: 85% FP → ~2%
- NestJS: 90% FP → 0%
- custom_patternsで残差のhelper delegationにも対応可能

4つのAIの意見を「材料」として使い、人間が組み合わせを決めた。どれか1つだけ採用していたら、この結果にはならなかった。

### 6. 結論: AIの使い方が変わった（300字）

- 1つのAIに聞くと「答え」をもらう感覚。疑う余地がない
- 4つに聞くと「選択肢」が見える
  - 今回の問いでは: GPTはリフレーミング、Claudeは構造化、Geminiは将来設計、Grokはユーザー体験にそれぞれ強かった
  - 別の問いでは違う傾向が出るかもしれない。大事なのは「複数の視点を得る」こと
- 4つに聞くコスト: 30分。設計ミスの手戻りコスト: 数日。安い投資
- 重要な設計判断ほど、単一のAIではなく複数AIに同じ問いを投げて比較する。それだけで判断の質が変わる
