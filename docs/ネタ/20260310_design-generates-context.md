# 複式簿記はB/Sのためにデザインされたわけではない

## 角度

デザインは文脈に従属するだけでなく、新たな文脈を立ち上げる力を持つ。

## 具体例

- 複式簿記: 取引記録の整合性管理 → B/S・P/Lという認知的枠組みを生成
- Git: 分散バージョン管理 → PR文化・協働の仕方を生成
- Unix pipe: プロセス間通信 → 「小さなプログラムを組み合わせる」思考様式を生成
- HTTP: 学術文書共有 → REST → マイクロサービスという組織構造を生成

## キーメッセージ

道具のデザインは目的を超えて、思考の枠組みそのものを作り変える。
デザインが認知的インフラストラクチャになる。

## 補強案

- dev-crewのTDDワークフローが「開発の仕方」自体を変えた実体験と接続
- GoFパターンがアレグザンダーの思想の核心を取りこぼした話（内部構造の定石集に還元）

## 注意

- 他者のClaude会話が起点なので、実体験の軸が弱い
- 自身の開発経験と接続して地に足をつける必要あり

## 推奨パターン

主張型（具体例から入る → 構造を抽出 → ソフトウェアに引き戻す）
