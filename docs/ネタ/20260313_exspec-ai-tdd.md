# 記事3: AIにAI用ツールをTDDで作らせた -- 2週間で681テスト、17ルール、4言語

## ターゲット読者

- Claude Code / AI駆動開発に興味があるエンジニア
- TDDを実践しているが、AIとの組み合わせ方を模索している人
- 「AIにどこまで任せられるか」を知りたい人

## 主張

AIは「作業」を代替する。人間は「判断」を代替しない。
AIにTDDで開発させると、人間の仕事は「コードを書くこと」から「何を作るか決めること」に変わる。
判断のボトルネックを人間に集中させることで、開発速度とコード品質を両立できる。

## 構成

### 1. 導入: 入れ子構造の面白さ（200字）

- AIが書くテストの品質をチェックするツールを、AIにTDDで作らせた
- テスト設計品質チェッカーのテストを、AIが書く。再帰的
- 人間がやったのは: 設計判断、優先順位、「次」を決めること

### 2. やり方: AIとのTDD壁打ち（500字）

特定のツールやフレームワークの話ではなく、AIとTDDを組み合わせる「やり方」として見せる:

**基本サイクル:**
1. 人間が「何を作るか」を決める（仕様を書く/口頭で伝える）
2. AIにテストを先に書かせる（失敗することを確認）
3. AIに最小限の実装をさせる（テストが通ることを確認）
4. AIにリファクタさせる（テストが通り続けることを確認）
5. 人間がレビューして「これでいいか」を判断

人間が介入するのは1と5だけ。2-4はAIが回す。

**壁打ちのコツ:**
- 「テストを先に書いて」と言うだけでAIはTDDを回せる
- 重要なのはAIに「判断」を委ねないこと。「どちらがいいと思う？」ではなく「Aでやって。理由はこう」
- テストが常に存在するから、AIの出力を信頼できる。テストなしだとAIの出力を毎回目視確認する必要がある
- 「テストなしでいいから急いで」は禁句。36サイクルでリグレッションゼロだったのはTDDのおかげ

### 3. 2週間で何が起きたか -- 人間が判断した3つの場面（800字）

数字: 2026-03-06 〜 2026-03-12。36サイクル。681テスト。17ルール。4言語。

数字だけでは伝わらないので、人間が実際に判断した場面を3つ具体的に見せる:

**場面1: AIが作った機能を削除する判断（3/7）**
- AIがT104（hardcoded-only）を実装した。テストも通った
- しかし、実プロジェクトで走らせたら「DAMP-style tests」（意図的にリテラルを書くテスト設計）を罰する
- AIに聞くと「ルールとしては正しい」と言う。でも「テスト設計の文脈ではDAMPは正当」と人間が判断
- T104を削除し、別のルールに置き換えた
- AIは「実装が正しいか」は判断できるが、「この機能を提供すべきか」は判断できない

**場面2: 精度は高いのにノイズが多い。下げるか？（3/10）**
- あるルールが全テストの36-48%に引っかかった。TP率は高い（本当に問題がある）
- でも量が多すぎて、重要度WARNだと実用にならない
- AIに聞くと「TP率が高いのでWARN維持が妥当」と言う。正しいが、ユーザー体験を見ていない
- 人間が「INFOに下げる。精度が高くてもノイズが多ければ使われない」と判断

**場面3: 1文字変更で594件の誤検知が消える。でもリスクは？（3/11）**
- `^assert_` → `^assert`。アンダースコア1文字の違い
- 「外すと偽陰性が増えないか？」AIは両方のリスクを列挙するだけ
- 「assertで始まるメソッド名でアサーションでないもの」を具体的に探させた
- 結果: 実害なし。1文字変更で594件の誤検知が消えた
- AIはリスクを列挙できるが、「このリスクは取れる」という判断は人間がする

### 4. AIが苦手だったこと（500字）

- **仕様の境界判断**
  - 実例: チェーンの深さをどこまで対応するか。AIは「どちらでもいい」と言う。実コードを調べて人間が決めた
- **「これは誤検知か？」の判断**
  - 実例: あるメソッドはアサーションか？ AIは「かもしれない」。ソースコードの用途を確認して人間が決めた
- **「テストなしで急いで」の誘惑を断る**
  - 実例: 大量の誤検知を見つけると「パターン追加だけしてテストは後で」と思う。毎回テストを先に書いた。結果、36サイクルでリグレッションゼロ

### 5. AIが得意だったこと（500字）

- **テストデータの量産**
  - 実例: 「このパターンを認識するテスト」を依頼したら、関連する4パターンを10秒で生成。人間が1つずつ書いたら20分かかる
- **パターンの横展開**
  - 実例: 1つの誤検知パターンを見せると「この形も同じ。これも。これも」と自動列挙。1つの問題から関連パターンを網羅する能力は人間より高い
- **安全なリファクタ**
  - 実例: 5箇所に散らばっていた定義を1箇所に集約。テストが通ることを確認しながら変換するので壊すリスクが低い
- **並列レビュー**
  - 実例: 3つの観点（正確性/パフォーマンス/セキュリティ）で同時レビュー。人間がやると30分。AIは3分

### 6. self-dogfoodingの再帰（300字）

- 作ったツールのテストを、そのツール自身でチェックする
- 最初は5件の指摘が出た。全部直した
- 今は指摘ゼロ
- 「自分のツールに自分が合格する」という説得力
- これはAIにやらせて初めて現実的。人間が手動でやるには面倒すぎる

### 7. 結論: AIに全部任せるのではなく、判断を人間に集中させる（300字）

- コードを書く時間: 2週間で合計2-3時間
- 判断する時間: 2週間で合計10-15時間
- 判断の内容: 「何を作るか」「何を直すか」「何を後回しにするか」「この設計でいいか」
- AIが作業を全部やることで、人間は判断に集中できる。判断の質が上がる
- ただしTDDがなかったら成立しない。テストがないとAIの出力を信頼できない
