# AI時代のテストアーキテクチャ: 3つのAIと壁打ちして設計した話

## 角度

Claude / Grok / Gemini の3者に同じ問いをぶつけて「AI時代のテストの理想形」を設計した実践記。
ML/データサイエンスドメインまで含めた「テスト = 仕様」を体系化する。

## 差別化

- 既存の「AI時代のテスト」記事はWebアプリ前提が多い。確率的ドメイン (ML, シミュレーション) にまで拡張した議論は少ない
- 3つのAIの回答を比較・統合する過程自体がコンテンツ。単一AIの出力ではない
- BDD / Clean Architecture 4層を「不採用」と明言し、理由を記録している。採用した話より不採用の理由の方が実務者には有益

## 構成案

1. **問題提起**: 119k行のテストコードがある。AIにリファクタさせたいが、そもそも「理想形」が定義されていない
2. **理論調査**: Executable Specification, Property-Based Testing, Metamorphic Testing の概念整理
3. **3AI壁打ち**: 同じ問いを Claude / Grok / Gemini に投げ、回答を比較。一致点と分岐点を整理
4. **結論**: 2領域モデル (決定論的 / 確率的) + Data Contract + Property-Based Testing に収束
5. **不採用リスト**: BDD (中間言語不要), 4層 (過剰), JSON parametrize (コンテキスト喪失)
6. **移行戦略**: 大規模リファクタしない。新規100% + ボーイスカウトルール
7. **AI時代の本質**: 「何が正しいか」の定義が人間の仕事。実装はAIに渡せる

## キーメッセージ

「AI時代のテストは、AIに『何を作れ』と伝えるための仕様書になる。テストを書く = 正しさを定義する = 人間の最後の仕事」
