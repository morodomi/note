# 記事案: Claude Codeのinsightsに自分を分析させたら、技術力より運用思想が露出した

## ターゲット読者

- Claude Code などのAIコーディングツールを日常的に使っている人
- AIに作業を委譲した結果、何がログに残るのか気になる人
- 自己分析、採用、評価にAIログ分析を使えるか考えたい人

## 主張

`/insights` が見ていたのは、書けるコードの種類よりも、仕事の進め方そのものだった。
開発ログをAIに読ませると、技術力の棚卸しより先に、プロセス設計、委譲スタイル、弱点の偏りが可視化される。

## 角度

- 「insights を試した」ではなく、「AIは成果物より先に働き方を読んでいる」を主軸にする
- 褒め分析よりも、弱点分析の刺さり方を本論に置く
- 自己分析ツールとしては面白いが、採用評価に直結させるのは危うい、という留保も入れる

## 見出し案

1. Claude Codeのinsightsに自分を分析させたら、技術力より運用思想がバレた
2. AIはコードだけ見ていない。開発ログから私の強みと弱点が可視化された
3. Claude Codeのinsightsを読んで驚いた。分析されたのは技術よりマネジメントだった

## 構成案

### 1. 導入

- 軽い気持ちで `/insights` を叩いた
- 技術スタック一覧や得意言語の話が返ると思っていた
- 実際に返ってきたのは、プロセス設計、委譲の仕方、品質管理の癖だった

### 2. まず驚いたのは「技術的な強み」の見方

- Rust/Python/TypeScriptを触っている、で終わらない
- 「構造化されたプロセス設計力」「並列オーケストレーション能力」「品質管理への執着」といった、仕事の運び方が抽出された
- AIがコードの巧拙より、運用の癖を先に見ている感じがあった

### 3. もっと面白かったのは「思考と人格」の分析

- システム思考、反証主義、制御と委譲のバランス感覚
- かなり当たっている部分もある
- 一方で、ログからこういう推定ができてしまうこと自体が少し怖い

### 4. 本論は「悪いところ分析」

- 暗黙知の外部化不足
- 委譲後の検証コストの過小評価
- プロセスへの信頼が実態確認を遅らせる
- どれも能力不足ではなく、強みの裏返しとして出てきたのが面白い

### 5. ここから分かったこと

- AIに作業を任せると、成果物だけでなく仕事の癖がログとして蓄積される
- そのログを読むと、技術力より先にマネジメントスタイルや弱点が見える
- 自己分析には使える
- ただし採用や評価に使うなら、かなり慎重であるべき

### 6. 結論

- `/insights` は「何ができるか」より「どう働いているか」を可視化するツールだった
- AI時代には、コードを書く能力だけでなく、AIとの協業ログそのものが職務経歴書に近づいていくのかもしれない
