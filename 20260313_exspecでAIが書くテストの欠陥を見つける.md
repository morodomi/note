# AIが書いたテストを静的解析したら、誤検知との戦いが始まった

AIにテストを書かせると、量は出る。

しかし、安心は出ない。

私のプロジェクトは、実装もテストもほぼAIに作らせている。942テスト関数。CIは全部緑。カバレッジも悪くない。最初は「もう十分では」と思っていた。

ところが仕様変更を入れようとしたとき、手が止まった。どのテストがどの仕様を守っているのか、ぱっと見で分からない。テスト名を見ても振る舞いが読めない。mockが多すぎて、内部実装を少し変えるだけで関係ないテストまで壊れそう。

テストは通る。ただ、仕様書としては弱い。

この違和感が、[exspec](https://github.com/morodomi/exspec)を作った動機だ。AIが書いたテストの「動く」ではなく、「ちゃんと仕様になっているか」を静的に点検するツール。

---

## カバレッジ100%でも、不安は消えない

既存の指標にも、それぞれ役割がある。

ただ、役割が違う。

- カバレッジは、どこを通ったかを見る
- mutation testingは、欠陥を検出できるかを見る
- コードレビューは、人間が文脈ごと読む

ここに、ひとつ穴がある。

「そのテストは仕様として読めるか」だ。

たとえば、関数を呼ぶだけでアサーションが1つもないテスト。CIは通る。カバレッジにも貢献する。ただ、そのテストが何を保証しているのかは分からない。

あるいは、mockを6個使って呼び出し順まで細かく見ているテスト。たしかに今の実装は守れる。しかし守っているのは振る舞いではなく内部構造だ。リファクタした瞬間に壊れる。

こういう問題は、数字だけ見ていてもなかなか出てこない。人間が読めば分かる。ただ、AIが1日に何十本もテストを書いてくるなら、毎回全部は読めない。

だったら、テストコードにもlintが要る。

---

## exspecは何を見るのか

exspecがやっていることはシンプルだ。テストコードをASTで解析して、「仕様として読みにくくなる構造」を拾う。LLMは使わない。APIコストもない。CIに入れやすく、同じコードには同じ結果を返せる。そこが大事だ。

今は主に、こんなものを見ている。

- アサーションがゼロのテスト
- mock依存が多すぎるテスト
- 長すぎるテスト
- 実装の詳細を直接なぞっているテスト
- 名前が曖昧で、何を検証しているか読めないテスト

テストの中身を採点しているというより、「このテスト、仕様として読むにはつらくないか」を構造から当てにいっている。

以前、[テストの理想形を言語化したらAIへの指示が変わった](https://note.com/morodomi/n/nf9e820194410) という記事で、良いテストの条件を4つに整理した。

- What not How: 実装ではなく振る舞いを見ているか
- Living Documentation: テスト名と中身が仕様書として読めるか
- Compositional: 1本のテストが1責務に収まっているか
- Single Source of Truth: 同じ仕様を重複して書いていないか

exspecは、この4つを完全に判定する道具ではない。ただ、それを壊しやすいパターンは見つけられる。そこに価値がある。

## 実際に走らせると、こういう出力になる

```bash
cargo install exspec
exspec .
```

出力はこんな感じだ。

```text
exspec v0.1.2 -- 8 test files, 42 test functions
BLOCK tests/test_user.py:12 T001 assertion-free
WARN  tests/test_user.py:45 T002 mock-overuse
WARN  tests/test_api.py:78 T003 giant-test
INFO  tests/test_api.py:120 T109 undescriptive-test-name
Score: BLOCK 1 | WARN 2 | INFO 1 | PASS 38
```

まず見るべきは「BLOCK」だ。ここは優先順位が明確だ。少なくとも「このまま放置しないほうがいい」場所が出る。

「WARN」は文脈次第だ。たとえばmockが多いのが悪ではない場面もある。ただ、何も見ずに流すには重い。レビュー対象としては十分だ。

「INFO」は棚卸し用だ。今すぐ直さなくてもいいけれど、テスト設計の癖は見えてくる。

この区分が大事で、最初から全部きれいにしなくていい。まず「BLOCK」だけ潰す。それだけでもかなり違う。

## dogfoodingで見えた、実際の誤検知

実例を1つ出す。

NestJSのdogfoodingで、最初「T001 assertion-free」が813件出た。かなり多い。正直、数字を見た瞬間は「さすがに壊れているのでは」と思った。中身を見たら、そのうち575件、つまり71%が supertest の「.expect()」をアサーションとして認識できていないだけだった。

たとえば、こういうテストだ。

```typescript
it('returns 200', async () => {
  await request(app.getHttpServer())
    .get('/health')
    .expect(200);
});
```

人間が見れば、これは明らかに何かを検証している。ところが当時のexspecは、関数としての expect(...) は認識できても、メソッドチェーン上の「.expect()」を拾えていなかった。

そこでパターンを1つ追加したら、NestJSの「T001 BLOCK」は「813 -> およそ238」まで減った。

そのあと、NestJS以外のdogfoodingも含めて、Chai の別名や property chain、Sinon の「.verify()」みたいな別系統の見逃しも潰した。再検証すると NestJS でも「238 -> 90 -> 17」まで下がった。残った17件は真の問題だった。

最初から完璧に判定するわけではない。実プロジェクトにぶつけると、どの書き方を見逃しているかが数字で出る。それを1つずつ潰していく。

だから「精度100%です」とは言わない。むしろ逆で、dogfoodingしながら信頼できる「BLOCK」を育ててきたツールだ。

## 自分のプロジェクトで見えたこと

最初に自分のプロジェクトで走らせた。942テスト関数。

結果は「BLOCK 0 | WARN 12 | INFO 23 | PASS 907」だった。全部ひどかったわけではない。むしろ大半は通る。それでも、十数件のWARNを追うだけで、AIがどういうときに雑なテストを書くかがかなり見えた。

公開前のdogfoodingでも傾向は似ていた。

- Laravelでは「T001 BLOCK 1305件」。Mockery対応後に「776件」まで減少
- Symfonyでは「17,148テスト」のうち「91件」が「$this->addToAssertionCount(1)」の見逃し

誤検知は確かにある。ただ、直せる誤検知はかなり機械的に潰せるし、潰したあとに残る「BLOCK」はだいぶ意味を持つようになる。

目立ったのは3つだ。

1つ目は、アサーションのないテスト。例外が出なければ通る、いわゆる「実行確認だけ」のテストだ。動作確認にはなっても、仕様の保証にはならない。

2つ目は、mockの使いすぎ。外部APIの境界だけならまだいい。ところが内部の依存関係まで全部mockしていると、振る舞いではなく実装手順をテストし始める。これが一番つらい。

3つ目は、巨大テスト。1本で何パターンも見ているので、失敗したときに何が壊れたか分からない。テスト名も曖昧になりがちだ。

面白かったのは、exspecの指摘をAIに見せると、AI自身が「たしかにこのテストは弱い」と認めることだ。人間が1本ずつレビューして説明するより速い。雑に言うと、AIが書いたテストを、別の機械で測っている。

## exspecを作った理由は、コードレビューが追いつかないから

この話は、単なるテストツールの紹介で終わらない。AIがコードもテストも量産するようになると、人間のレビュー速度がボトルネックになるからだ。

実装コードはlintや型でかなり仕組み化できるのに、テストコードだけ「人間が読めば分かる」で止まっていた。そこを埋めたかった。

私の感覚では、役割分担はこうなる。

- 実装コードはlint、型、テスト、カバレッジで落とす
- テストコードはexspecのような構造チェックで落とす
- それでも残る「本当にこの仕様でいいか」だけを人間が判断する

レビューを全部捨てるわけではない。ただ、人間が目で読む前に、機械で落とせるものは先に落としたほうがいい。exspecはそのための部品だ。

## 制約もかなりある

ここは正直に書いておく。

exspecはまだ若い。誤検知もある。プロジェクト固有のアサーションヘルパーは、そのままだと認識できないことがある。日本語のテスト名も、まだきれいには扱えない。Rustマクロ内の検出にも制約がある。

つまり、exspecが「良いテストかどうか」を最終判定するわけではない。

やっているのは、構造的に危ない場所を先に炙り出すことだ。完璧な判定器ではなく、レビューの入口を絞るための計測器に近い。

この割り切りは大事だと思っている。人間の代わりを全部やるツールではない。ただ、「BLOCK」を起点に見る用途なら、レビューの入口をかなり絞れる。

## まずは自分のリポジトリで1回走らせればいい

言いたいことは、これに尽きる。

AIにテストを書かせているなら、実装コードだけでなくテストコードも点検したほうがいい。カバレッジが高いことと、仕様として読めることは別だ。

```bash
cargo install exspec && exspec .
```

最初はこれだけで十分だ。「BLOCK」が0件なら、最低限の地雷は避けられている可能性が高い。もし出たら、そこが最初の改善ポイントだ。

942件のテストが全部緑だった日より、WARN 12件を潰した日のほうが、自分のテストを信頼できるようになった。数字が見えると、次に何をすべきかも見える。

GitHubはこちらだ。まずREADMEの出力例を見て、気になったら「docs/SPEC.md」と「docs/dogfooding-results.md」を見るのが早い。
