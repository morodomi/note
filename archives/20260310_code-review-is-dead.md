# コードレビューが死んだので、製造業の品質保証に乗り換えた

コードレビューは死んだ。

大げさじゃなく、少なくとも私の開発では、もう成立していない。

AIにコードを書かせると、同時に複数プロジェクトを回せる。仕様書を書いて渡せば、テスト作成、実装、リファクタリング、レビューまで一気に走る。生成が速すぎて、人間がリリースのボトルネックになる。Pull Requestが積み上がる。読みきれない。

これは私だけの話じゃない。OSSの世界では既に破綻が始まっている。

無限キャンバスSDKのtldrawは、低品質なAI生成PRの流入に耐えきれず、外部コントリビューターからのPRを自動クローズし始めた。ターミナルエミュレータのGhosttyは、承認済みissue以外のAI利用PRを即クローズ、質の悪いAI利用者は将来の貢献を全面禁止する方針を敷いた。どちらもメンテナが疲弊した末の判断。

コードの生成速度に、人間のレビュー速度が追いつかない。

## 「人間が見れば安心」は幻想

「でも人間がチェックしないと危ないでしょ」と言いたくなる気持ちはわかる。

じゃあ聞くけど、人間がレビューすれば品質は担保されるのか。

されない。人による。

AIに任せるとスコープが広すぎたり狭すぎたりするし、テストがカバーしきれないこともある。じゃあ人間なら完璧か。そんなわけがない。100%のカバレッジなんて、人間でも無理。

漏れは仕様書の段階でカバーしておくべきで、漏れたらリリースし直すしかない。それは人間が書いたコードでも同じこと。

AI生成コードにはAIなりの癖もある。もっともらしい誤実装、既存設計との不整合、テストごとまとめて間違う共犯関係。そこは注意が要る。ただ、問題の本質はリスクの種類より量だと思っている。

1つのPull Requestに時間をかけて、完璧に仕上げてからリリースする。それ自体は正しい。ただ、AIが1日に10本のPull Requestを出してくる世界で、その正しさを維持できるか。正直、無理。現場にもよるけど、少なくとも個人開発やスタートアップでは、本番データを壊さない設計にしておいて、壊れたらログで検知して直す方が速い。

## コードを「読む」から「測る」へ

じゃあレビューの代わりに何をするか。

私がやっているのは、LLMを使わない静的解析と統計解析の組み合わせ。Lint、PHPStanレベル8〜9、コードカバレッジ80〜90%。曖昧さがない。数値で出る。

ここが大事で、LLMに「このコード大丈夫？」と聞くのは、レビューの自動化に見えて、実はレビューの問題をそのまま引き継いでいる。LLMの判断には曖昧さがある。「問題なさそうです」は、人間の「たぶん大丈夫」と変わらない。

製造業の品質保証を考えてみてほしい。

工場では製品を1個ずつ職人が目視チェックしているわけじゃない。計測器で全数検査して、基準を満たしているかを数値で判定している。寸法が公差内か。強度が規格値以上か。人間の感覚ではなく、機械が判定する。

ソフトウェアのLintやコードカバレッジは、まさにこれと同じ発想。「コードを読んで良し悪しを判断する」のではなく、「基準を決めて、満たしているかを機械的に判定する」。

もちろん、ソフトウェアは製造業と同じではない。仕様の妥当性、ビジネスとの整合性、本当にその変更が必要かどうか。こういう判断は測れない。測れるものと測れないものがある。だからこそ、測れるものは機械に任せて、人間は測れないものに集中すべきだ。

## テストコードの品質保証が次の問題

AIにテストを書かせて、Lintでコード品質をチェックして、カバレッジで最低限の検査漏れを減らす。この流れは筋が通っている。

でも、ここで1つ問題が出てくる。

テストコードもAIが書く。テストコード自体の品質保証が宙に浮く。

テストが通っているからOK、ではない。テストの構造自体が甘ければ、カバレッジの数字は高くても、実際には何も保証していない。形だけのテストを量産して、緑のチェックマークが並ぶ。見た目は安心。中身はスカスカ。

たとえば、こういうテストをAIは平気で書く。

```python
def test_create_user():
    user = create_user("alice", "alice@example.com")
    # assertが1つもない
```

関数を呼んでいるだけ。例外が出なければ通る。カバレッジには貢献する。しかし、何も検証していない。

あるいは、モックを5個も6個も使って、テスト対象の実装の内部をそのままなぞっているテスト。振る舞いではなく実装を検証している。リファクタリングするたびに壊れる。テストの意味がない。

こういう構造的な問題は、カバレッジの数字には表れない。人間がテストコードを読めばわかるけど、AIが1日に数百のテストを生成する世界で、全部読むのか。結局、同じ問題に戻ってくる。

## テストコードにもLintが要る

この問題に対して、今exspecというツールを作ってみている。

やっていることは単純で、テストコードを静的解析して、構造的な品質を数値でチェックする。LLMは使わない。

たとえば、こういう指標を見ている。

- アサーションがゼロのテスト関数はないか
- モックが多すぎないか（5個以上は警告）
- テスト関数が長すぎないか（50行超えは警告）
- テスト関数あたりのアサーション数は十分か
- パラメタライズドテストが使われているか

どれも、コードを「読んで」判断しているわけじゃない。ASTを解析して、数を数えて、閾値と比較しているだけ。1ファイルの解析に1-5ミリ秒。100ファイルのプロジェクトでも1秒かからない。APIの呼び出しはゼロ。

考え方は同じ。基準を決めて、満たしているかを機械が判定する。それだけ。

今は自分のプロジェクトでdogfoodingしながら、誤検知を潰している段階。閾値の設定が甘いと、正当なテストまで警告する。そこの調整が地味にきつい。

## LLMレビューが解決策にならない理由

「LLMにレビューさせればいいじゃん」という反論は当然ある。

実際、私もdev-crewの中でAIレビューを組み込んでいる。正確性、パフォーマンス、セキュリティ。複数の観点で並列レビューをかけている。

ただ、これはレビューの自動化であって、品質保証ではない。

LLMのレビューは「たぶん大丈夫だと思います」の高級版。確率的に正しいけど、決定論的ではない。同じコードを2回見せたら、違う指摘が返ってくることがある。再現性がない。

静的解析は違う。同じコードには、必ず同じ結果を返す。閾値を超えたら警告。超えなければ通過。曖昧さがゼロ。

だから、LLMレビューと静的解析は対立するものじゃなくて、レイヤーが違う。

静的解析で「測れるもの」を先にふるいにかけて、残った「判断が必要なもの」だけをLLMか人間がレビューする。全部を人間やLLMに投げるのは、工場で全製品を目視検査するようなもの。最初に計測器を通せば、目視が必要な製品は激減する。

## レビュー文化の終わり方

コードレビューがなくなると言っても、明日から全員がレビューをやめるわけじゃない。

たぶん、こうなる。

個人開発者はもうレビューを捨て始めている。テストとLintで品質を担保する方が、速くて合理的だから。小規模チームも時間の問題。Pull Requestの承認フローを省略して、CI/CDに品質ゲートを組み込む方が回る。大企業は最後まで残るだろうけど、AIが書くコードの量が増えるにつれて、形骸化は避けられない。

人間が見るのは仕様書とビジネスインパクトだけになる。コードそのものは、テストと静的解析が保証する。属人的な判断から、再現可能な仕組みへ。

製造業が数十年前にやったことを、ソフトウェア開発がようやくやる。遅すぎたぐらいだ。
