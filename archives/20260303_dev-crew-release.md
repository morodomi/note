# 暴走するAIを飼い慣らすまで9ヶ月。dev-crewを公開する

前回、「10個の自作スキルを捨ててClaude Codeのデフォルトに戻した」と書いた。

捨てたのは運営だ。コードは残っている。そのコードを整理して、1つのプラグインにまとめた。dev-crewという名前で公開した。

https://github.com/morodomi/dev-crew

33エージェント、29スキル、3ルール。人間の開発チームと同じ構造をClaude Codeの中に作った。PdM、エンジニア、デザイナー、セキュリティ監査、レビュアー、メタラーナー。6つの職種がTDDサイクルを回す。

この記事では、dev-crewが今の形になるまでの9ヶ月を書く。

---

## 全部やらせようとして、爆発した

始まりは去年の7月だった。

Claude Codeが出たばかりで、何でもやらせてみたかった。設計も実装もテストもデプロイも、全部AIに任せた。自分は承認ボタンを押すだけ。1日で何十ファイルも生成されて、プロジェクトが膨れた。

何が起きているか、わからなくなった。

AIが書いたコードをAIにレビューさせて、AIが修正する。自分は「OK」を押しているだけ。コードの中身を理解していないから、エラーが出ても原因がわからない。AIに聞くと「直しました」と返ってくる。本当に直ったのかもわからない。

あの頃の記事にこう書いた。「私はただの承認ボタンだった」。

---

## 首輪を作った。すぐ抜けた

次に考えたのは、AIに制約を課すことだった。

ait3というnpmパッケージを作った。CLAUDE.mdに制約ルールを書いて、AIの行動範囲を縛る。「このディレクトリ以外は触るな」「テストなしでコミットするな」。

2週間で破綻した。

AIは指示を守るように見えて、抜け道を見つける。制約が増えるとコンテキストを圧迫して、肝心の作業精度が落ちる。制約を管理する手間が、本来の開発より重くなった。

首輪は解決策じゃない。「AIの首輪はすぐ抜ける」という記事を書いて、このアプローチを手放した。

---

## Claude Codeの進化に乗る

ここから先は、Claude Code本体の進化に合わせて自分のやり方を変えていった。

**commands時代**。Claude Codeにcommands機能が追加された。手動でTDDフローを追えるようになった。planを書いて、testを書いて、実装して、リファクタ。1つずつコマンドを叩く。手動だけど、フローが安定した。暴走しなくなった。

**skills時代**。skills機能が追加された。commandsをまとめて、連続実行できるようになった。ここでtdd-skillsとredteam-skillsが生まれた。TDDサイクルの自動化と、セキュリティスキャンの並列実行。手動のcommands時代から、半自動になった。

**Plan mode時代**。Plan modeが追加された。Claude Codeがコードベースを探索して、設計方針を立てて、承認を求める。承認したらタスク完了まで自走する。自分のplanスキルが不要になった瞬間だ。

そのたびにスキルを捨てて、作り直した。Claude Codeの機能で代替できるものは標準に任せて、標準にないものだけを自作する。ずっとこれをやっている。

---

## スキルを捨てながら作った

tdd-skillsには、コアのTDDフローと言語別のサブスキル（tdd-php、tdd-python、tdd-tsなど）があった。redteam-skillsにはセキュリティスキャン。meta-skillsには自己学習。それぞれ別のプラグインだった。

dev-crewは、これらの設計を1つに統合したプラグインだ。2月14日から3月3日まで、約18日間で81コミット。

統合する過程で、dev-crew自身のスキルも削った。リファクタリング工程に自前のスキルを作り込んでいたが、/simplifyがアップデートされて不要になった。品質チェック、重複検出、命名改善、構造整理。自分で定義していた観点を、/simplifyが1コマンドでカバーした。前回の記事で「10個の自作スキルを捨てた」と書いたのは、この話だ。

作りながら捨てる。Claude Codeの進化に追いつかれたスキルは、その場で消す。作っては壊す。dev-crewの開発自体がそれだ。

---

## dev-crewの全体像

核になっているのはTDDサイクルだ。

```
plan mode → init → kickoff → red → green → refactor → review → commit
```

plan modeで設計方針を決める。initでコンテキストを設定する。kickoffでCycle docを生成して、plan modeからaccept editsモードに切り替える。ここが境界だ。設計フェーズから実装フェーズに入る。redでテストを書く。greenで最小限の実装をする。refactorで/simplifyに委譲する。reviewで複数エージェントが並列レビューする。commitで変更をGitに記録する。

Cycle docは、サイクル全体の設計書であり記憶装置だ。Claude Codeのコンテキストウィンドウには限りがある。長いセッションでは過去の会話が圧縮されて、サイクルの設計意図が消える。Cycle docにフェーズごとの判断と結果を書き残しておけば、コンテキストが圧縮されても読み直せる。

Claude Code本体にもコンテキスト管理の機能はある。ただ、TDDサイクルに必要な情報だけを、必要な粒度で残すには、自前で定義したほうが良かった。必要以上の情報は要らない。

このサイクルを6つの職種が回す。PdMがCycle docを生成して優先順位を決める。エンジニアがred/green/refactorを実装する。レビューは正確性、パフォーマンス、セキュリティ、ユーザビリティの並列チェック。セキュリティ監査は別軸だ。SQLインジェクション、XSS、認証バイパスなど13種類の攻撃エージェントで脆弱性を叩く。メタラーナーはセッションからパターンを抽出して、スキルを自己改善する。

デザイナーまで入れたのはやりすぎだったかもしれない。ただ、UIの判断を毎回自分でやるのが面倒だった。必要な職種だけが、必要なタイミングで動く。全部が1つのプラグインに収まっている。

## なぜ「組織」なのか

AIに「このコードをレビューして」と言うと、「問題ありません」と返ってくる。あるいは指摘が返ってきても、それが正しいのかわからない。プロンプトをどう書けば正しさを保証できるのか、答えがない。

人間が最終判断を下すとき、必要なのは情報の確からしさだ。ビジネスの視点、デザインの視点、セキュリティの視点、インフラの視点。それぞれの専門領域から見た評価がないと、判断の根拠がない。

人間の組織でも同じだ。セキュリティ担当が「この入力のバリデーションが甘い」と言い、パフォーマンス担当が「このクエリはN+1になる」と言う。特定の視点からの具体的な指摘があるから、受け入れるかどうか判断できる。「全部見ました、大丈夫です」だけでは信用できない。

dev-crewもこの構造にした。セキュリティエージェントにはセキュリティだけを見させる。パフォーマンスエージェントにはパフォーマンスだけを見させる。ビジネス判断はPdMに、ユーザビリティはデザイナーに。それぞれの視点から根拠付きで報告させる。最終的にはPdMなり自分なりが、専門エージェントの報告を総合して判断を下す。

1つの万能なAIに全部見させるのではなく、専門性を持ったエージェントの合議で全体をカバーする。個々のAIは専門家でしかないが、束ねれば単一のエージェントでは見落とす問題を拾える。今のAIの能力で正しさを担保するには、これが現実的だ。

## なぜ公開したのか

tdd-skillsもredteam-skillsも、GitHubに公開していた。

tdd-skillsはメンテしなかった。2週間で考え方が変わる。古いコードが残っているのに、スターだけ増えていく。古いまま公開しておいて馬鹿にされるのも嫌だった。かといって2週間おきにリポジトリを作り直すのも意味がない。

結局、放置した。

dev-crewを公開しようと思ったのは、Xを見ていたときだ。自分が1週間前に実装していた内容と同じことを、今バズっている投稿がある。意外とみんなキャッチアップ大変なんだなと思った。

自分だって毎日コツコツやっているだけだ。たまたまClaude Codeを去年の7月から使い続けていて、進化に合わせてツールを作り替えてきただけだ。

dev-crewは去年7月からの9ヶ月の集大成になっている。tdd-skillsと違って、今の最新のやり方が入っている。だから公開してもいいかと思った。

---

## メンテするとは約束しない

dev-crewは気が向いたらメンテする。ただし、issueやPRは受け付けない。

理由は2つ。

まず、Claude Codeの進化が速すぎる。2週間でやり方が変わる。外部のissueに対応している間に、前提が変わる。

もう1つ。価値の本体はコードじゃない。コードは思想の実装にすぎない。思想はこのnoteに書く。「なぜTDDを強制するのか」「なぜレビューを並列化するのか」「なぜセキュリティスキャンをサイクルに組み込むのか」。個別の機能は別記事で書いていく。

リポジトリを見て「参考にした」「フォークして改造した」は歓迎する。ただし、一緒に育てるつもりはない。

---

## tokenという現実

理想を言えば、毎回セキュリティチェックをかけたい。並列化できるエージェントは全部並列に走らせたい。13種類の攻撃エージェントを毎コミットで回して、レビュアーは5観点を同時に走らせて、メタラーナーが毎セッション学習する。

現実は違う。

tokenが爆発する。Maxプランのx20でも、全部を毎回回したら3日で上限に達する。人間のレビューも追いつかない。5つのレビューエージェントが同時にレポートを返してきたら、全部読んで判断する時間がない。

だからバランスを取っている。セキュリティスキャンはリリース前だけ。レビューは変更規模に応じてエージェント数をスケーリングする。メタラーナーは週1回。

理想と現実のギャップはある。でも、理想の形を定義しておくことに意味がある。tokenが安くなれば、あるいはモデルが効率化すれば、dev-crewの構成を変えずに精度を上げられる。
