# TDDをAIに任せる、Agent Teamsで変わったこと

Agent Teams を試したら、3時間で Max 20x プランの usage を使い切った。

意思決定の質は確かに上がった。でも燃費が悪すぎる。これは、AIにTDDを回させる仕組みを Agent Teams に載せ替えてみた1日目の記録です。

## tdd-skills とは

[tdd-skills](https://github.com/morodomi/tdd-skills) は、Claude Code に TDD を強制するプラグインです。INIT → PLAN → RED → GREEN → REFACTOR → REVIEW → COMMIT の7フェーズを自動で回します。各フェーズには専門エージェントがいて、設計レビューや品質レビューも並列実行します。

今回の話は、この tdd-skills の内部アーキテクチャを Subagent から Agent Teams に載せ替えた記録です。

## Subagent時代の課題

tdd-skills では、quality-gate（品質レビュー）を Subagent（結果を返すだけの並列処理機能）で実行していました。6つのレビューエージェントが並行して動く仕組みです。

各エージェントがレビュー結果を返して、メインのエージェントがそれを統合する。

その仕組みで動いてはいました。ただ、エージェント同士が議論できない。あるレビュアーが問題を指摘しても、別のレビュアーはその指摘を知らないまま判断する。情報が分断されたまま、メインのエージェントが1人で統合判断していました。

## Agent Teams に載せ替えた

やったことは3つです。

1. plan-review（設計レビュー）を Agent Teams 化
2. quality-gate（品質レビュー）を Agent Teams 化
3. RED、GREEN などの worker も Agent Teams 化

設定は `settings.json` に環境変数を追加するだけです。

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

tdd-skills 側でこの環境変数を自動判別します。設定がなければ従来の Subagent のまま動くので、移行リスクはありません。

Agent Teams モードでは、以下のように動きます。

- **PLAN フェーズ**: architect エージェントが設計とテストリスト作成を担当
- **plan-review**: 複数のレビュアーが Agent Teams として議論しながらレビュー
- **RED/GREEN/REFACTOR**: 専門の worker エージェントが実装を担当
- **quality-gate**: 6エージェントが並行レビュー。エージェント同士で指摘を共有

全体を tdd-orchestrate が PdM として統括します。

## 良かったこと: 意思決定の質が上がった

本当に変わったのは「意思決定の質」です。判断待ちの回数が減った分、体感の処理速度も上がりました。ただし usage の消費も早い。

Subagent 時代は問題がありました。あるレビュアーが「この設計は危険」と指摘する。でも別のレビュアーはそれを知らず「問題なし」と返す。メインのエージェントが全結果を受け取って、1人で総合判断する。見落としが起きやすい構造でした。

Agent Teams では、tdd-orchestrate がプロダクトマネージャー（PdM）の役割を担います。専門エージェントに仕事を委譲して、返ってきた結果をもとに次のフェーズに進むかを決定する。

レビュアー同士がメッセージをやり取りできます。ある警告を踏まえて、別のレビュアーが意見を修正する。1人で抱え込まない分、精度が上がります。

tdd-orchestrate がある程度の判断をしてくれるので、放置しても開発が回ります。各フェーズを自律的に進めてくれる。

さらに、各エージェントが独立したコンテキストを持っています。そのため PdM 役のコンテキストが埋まりにくい。セッションが長持ちするのは副産物でした。

## 正直なデメリット: usage が増えた

正直に書きます。usage が増えるのはデメリットです。

私は Max 20x プランを契約しています。いつも4時間ほどで使い切っていました。

Agent Teams に切り替えたら、3時間で使い切りました。

理由は明確です。Agent Teams は各チームメイトがそれぞれ独立した Claude のインスタンスです。

Subagent は結果をサマリーにして返します。Agent Teams は違う。各エージェントがフルのコンテキストウィンドウを消費する。

並列で動くエージェントの数だけ、token が増えます。

[公式ドキュメント](https://code.claude.com/docs/en/agent-teams)にもこう書いてあります。

> Agent teams use significantly more tokens than a single session. Each teammate has its own context window, and token usage scales with the number of active teammates.

知ってはいました。でも実際に使うと結構きつい。

Opus 4.6 になった影響もゼロではないかもしれません。ただ、主因は Agent Teams の並列実行による絶対量の増加だと考えています。

## それでも使い続ける理由

usage が増えるのは痛い。でも、判断精度が上がっている実感があります。

Subagent 時代、quality-gate で「認証ロジックにセキュリティリスクあり」と警告が出たことがありました。でも別のレビュアーは「パフォーマンス問題なし」と返す。メインのエージェントは「総合的にOK」と判断した。その危険性が見落とされたんです。

Agent Teams では、セキュリティの警告が出た時点で他のレビュアーにも共有されます。PdM 役が「この件、他のレビュアーはどう思う？」と振ることもできる。見落としが構造的に減ります。

Subscription なので、使い切ったところで追加費用はかからない。ただ、5時間ウィンドウのリセットを待つ時間が増えた。コーヒーを淹れる回数が増えたのは、メリットかもしれません。

---

## まとめ

Agent Teams を1日使ってみた感想です。

**良かったこと**
- エージェント同士の議論で意思決定の質が上がった
- コンテキストが埋まりにくく、セッションが長持ちする
- 指摘の見落としが構造的に減る
- ある程度放置しても PdM に任せて開発が回る

**正直なデメリット**
- usage は確実に増える。Max 20x プランで1時間早く使い切った
- ウィンドウのリセット待ちが増える

AIを1匹の猟犬として使っていた頃は、燃費もそこそこだった。Agent Teams にしたら群れになった。群れは速い。獲物も確実に仕留める。ただし、エサ代は群れの分だけかかる。

燃費は悪い。でも見落としによる手戻りを考えれば、トータルでは安い。私は使い続けます。

設定は1行です。
