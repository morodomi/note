# コンテキストウィンドウは制約じゃない。フェーズの区切りだ

前回、dev-crewを公開した記事で「Cycle docは記憶装置だ」と書いた。そして「tokenが爆発する」とも書いた。

今回はその続き。コンテキストウィンドウが溢れる問題をどう解決したか。溢れないようにするのをやめた。溢れる前に書き出して、忘れて、必要になったら読み直す。

---

## AIは忘れる。しかも突然

Claude Codeのコンテキストウィンドウには上限がある。上限に近づくとauto-compactが自動で走って、古い会話を要約して圧縮する。問題は、このタイミングを自分で操作できないことだ。

TDDサイクルを考えてほしい。

```
plan mode → init → kickoff → red → green → refactor → review → commit
```

7つのフェーズがある。各フェーズで設計の議論があり、テストコードがあり、実装コードがあり、レビュー結果がある。フェーズが進むにつれてコンテキストは膨らんでいく。

問題は、5フェーズ目あたりで起きる。greenフェーズで実装しているときに、initフェーズで決めた設計方針が要約されて消える。「なぜこのアーキテクチャにしたのか」「なぜこのライブラリを選んだのか」。会話の冒頭で決めたことが、もうコンテキストにない。

厄介なのは、AIが「忘れました」と言わないことだ。設計方針が消えたコンテキストの中で、AIは残っている情報から勝手に判断して自走する。承認した設計とは違う方向に実装が進んでいく。聞けば「こちらの方が適切だと判断しました」と返ってくる。暴走の一種だ。

これが何度か起きた。最初は「コンテキストが足りないから」と思って、重要な情報を何度も貼り直していた。手動で。フェーズが変わるたびに「ここまでの設計方針はこうで、テストはこう書いて、実装方針はこうだ」と繰り返す。

馬鹿馬鹿しくなった。

---

## 会話で記憶するな。ファイルに書け

解決策は単純だった。会話に頼るのをやめて、ファイルに書く。

dev-crewにはCycle docという仕組みがある。TDDサイクル1回分の設計書兼記憶装置で、docs/cycles/ ディレクトリにMarkdownファイルとして置く。

```markdown
---
feature: user-authentication
cycle: 20260305_1430_user-auth
phase: GREEN
created: 2026-03-05 14:30
updated: 2026-03-05 16:45
---

# User Authentication

## Scope Definition
### In Scope
- [ ] ログインAPI実装
- [ ] セッション管理
### Out of Scope
- OAuth連携 (Reason: 次サイクルで対応)

## Environment
- Layer: Backend
- Plugin: flask
- Risk: 35 (WARN)

## Test List
### TODO
- [ ] TC-03: セッション期限切れテスト
### DONE
- [x] TC-01: 正常ログインテスト
- [x] TC-02: パスワード不一致テスト

## Implementation Notes
### Goal
メールアドレスとパスワードでログインできる
### Design Approach
Flask-Login + bcrypt。セッションはRedisに保存。

## Progress Log
### 2026-03-05 14:30 - INIT
- Cycle doc created
### 2026-03-05 15:00 - KICKOFF
- Plan review PASS
### 2026-03-05 15:30 - RED
- 3 tests created, all failing
```

フェーズが進むたびに更新される。会話のコンテキストが要約されても、このファイルは消えない。ディスクに永続化されているからだ。

ポイントは、「何を覚えているべきか」が構造化されていること。スコープ、環境、テストリスト、設計方針、進捗ログ。会話履歴のように全部を持つのではなく、TDDサイクルに必要な情報だけを、必要な粒度で残す。

---

## フェーズ境界で書き出す

Cycle docがあるだけでは不十分だった。「いつ書き出すか」が問題になる。

最初は各フェーズの最後に手動で「ここまでの結果をCycle docに書いて」と指示していた。忘れることがあった。書き出しを忘れたまま /compact を実行して、次のフェーズでCycle docが古いまま走り出した。AIは古い設計方針で実装を始めて、30分が無駄になった。

phase-compactというスキルを作った。フェーズの境界で自動的に呼ばれて、そのフェーズの成果をCycle docに書き出す仕組みだ。

書き出す内容はフェーズごとに違う。

- plan mode → KICKOFF: スコープ、環境、目標
- KICKOFF → RED: テストリスト、設計決定
- RED → GREEN: テストファイルのパス、失敗内容
- GREEN → REFACTOR: 実装ファイルのパス、テスト結果
- REFACTOR → REVIEW: リファクタ後のファイル、変更概要
- REVIEW → COMMIT: レビュースコア、指摘事項

各遷移で、Phase Summaryという形式でCycle docに追記する。

```markdown
### Phase: RED - Completed at 15:30
**Artifacts**: tests/test_auth.py, tests/test_session.py
**Decisions**: pytest + Flask test client, 3 tests created, all failing
**Metrics**: line_count=87, file_count=2, test_count=3
**Next Phase Input**: test files on disk, implement to make them pass
```

たった4行。前のフェーズで何を作ったか、何を決めたか、次のフェーズは何をすればいいか。これだけあれば、コンテキストが完全に消えても復元できる。

なぜ「何を保存するか」をフェーズごとに定義したのか。汎用にすると、何を保存するか毎回AIに判断させることになる。その判断が不安定だから困っているのに、保存の判断まで委ねてどうする。RED → GREENの遷移ならテストファイルのパスと失敗内容だけ。GREEN → REFACTORなら実装ファイルのパスとテスト結果だけ。フェーズごとに保存内容を決めておけば、判断の余地がない。決定論的に動く。

## 忘れさせる

phase-compactの次のステップは「忘れさせる」。

Claude Codeには /compact というコマンドがある。会話履歴を要約して圧縮する。過去の詳細なやりとりが短い要約に置き換えられて、コンテキストウィンドウに空きができる。

phase-compactは、Cycle docへの永続化が完了した後に /compact の実行を促す。

```
Phase Summary をCycle docに追記しました。
コンテキストを圧縮するため /compact を実行してください。
```

なぜ自動で実行しないのか。できないからだ。/compact はClaude CodeのCLI内部コマンドで、2026年3月時点ではプログラムから直接呼べない。だからユーザーに実行を頼む。

ここに設計思想がある。phase-compactは「重要な情報をファイルに書き出す」という自分の責務だけを果たす。「会話を圧縮する」というClaude Code本体の責務には手を出さない。単一責任の原則そのものだ。

---

## 読み直す

/compact を実行した。コンテキストは圧縮された。

ただ、圧縮された状態ではCycle docの内容も会話から消えている。次のフェーズを始める前に、Cycle docを読み直す必要がある。

reloadというスキルを作った。

```
================================================================================
Context Reload Complete
================================================================================
Cycle doc: docs/cycles/20260305_1430_user-auth.md
Phase: GREEN
Scope: ログインAPI + セッション管理
Artifacts loaded: 2 files
Next action: テストを通す実装を開始
================================================================================
```

reloadがやることは4つ。

1. 最新のCycle docを見つける
2. 全文読み込んで、現在のフェーズと成果物を把握する
3. Phase Summaryに記載されたファイル（テストコード、実装コードなど）を読み込む
4. 次にやるべきことを表示する

コンテキストが完全に消えた状態から、TDDサイクルの途中に復帰できる。セッションを跨いでも動く。昨日の作業の続きを今日やるとき、/reload 一発で前回の状態に戻れる。

## plan mode → approve → auto-compact

dev-crewのTDDサイクルは、plan modeから始まる。

plan modeはClaude Codeの機能で、AIがコードベースを探索して設計方針を立てて、人間が承認する。承認するとplan modeを抜けて通常の実行モードに入る。

この「plan modeの承認」が、自然なフェーズ境界になっている。Claude Codeの仕様で、plan modeからaccept editsモードに切り替わるタイミングで自動的にcompactが走る。

設計を議論している間にファイルを何十個も読んで、アーキテクチャを議論して、テストリストを作る。承認した時点でそれらが圧縮される。議論の詳細はもう要らない。結論はplanファイルとCycle docに書いてある。

冒頭で書いた通り、auto-compactはタイミングを操作できない。コンテキスト使用率が閾値を超えたら勝手に走る。CLAUDE_AUTOCOMPACT_PCT_OVERRIDEという環境変数で閾値を変えられるが（例えば80を設定すると80%超過で発動）、「今このタイミングで圧縮したい」という指定はできない。

phase-compactは、このタイミングの問題を解決する。

- auto-compact: コンテキスト使用率でトリガー。自動的な会話圧縮。タイミング制御不可
- phase-compact: フェーズ境界でトリガー。意図的な永続化 + 手動圧縮。タイミング制御可

phase-compactでCycle docに書き出してから、手動で /compact する。重要な情報が永続化された状態で圧縮が走る。auto-compactに任せると、永続化する前に圧縮されて情報が失われるリスクがある。だからフェーズ境界で自分のタイミングで圧縮する。

## 実際のフロー

具体的にどう動くか。ユーザーの視点で書く。

**1. plan modeで設計する**

Claude Codeがコードベースを探索して、設計方針を提案する。テストリストを作る。自分は「それでいい」「ここは変えて」と返す。設計が固まったら承認する。

この時点で、planファイルに設計の結論が記録されている。

**2. kickoffでCycle docを生成する**

planファイルからCycle docが自動生成される。スコープ、環境、テストリスト、設計方針がCycle docに転記される。

**3. redでテストを書く**

テストを書き終わったら、phase-compactが走る。テストファイルのパスと失敗内容をCycle docに記録して、「/compactを実行してください」と案内が出る。

/compact を実行。コンテキストが圧縮される。

**4. /reload で復帰する**

「/reload」と打つ。Cycle docとテストファイルが読み込まれて、「次はgreenフェーズ。テストを通す実装を始めてください」と表示される。

**5. greenで実装する**

テストが通る最小限の実装をする。phase-compactが走る。/compact する。

**6. 以降繰り返し**

refactor → review → commit。各フェーズでphase-compact → /compact → /reload のサイクルが回る。

## 面倒じゃないのか

面倒だ。

フェーズごとに /compact と /reload を手動で打つ。2回のキーストロークがフェーズごとに増える。

ただ、コンテキストが溢れてAIが設計を忘れる方がもっと面倒なんだ。「さっき承認した設計を覚えていますか？」「テストリストの3番目は何でしたか？」。こういうやりとりが発生するたびに、時間とトークンが無駄になる。

トレードオフの話だ。

- 何もしない: コストゼロ。ただしコンテキスト溢れで設計が消える
- 手動で情報を貼り直す: コスト高い（毎回）。貼り忘れ、貼り間違いのリスク
- phase-compact + reload: コスト低い（コマンド2回/フェーズ）。Cycle docの更新漏れリスクは構造化で軽減

完璧じゃない。3つの中では一番マシだ、というだけ。

Cycle docは残り続ける。コンテキストウィンドウがどれだけ大きくなっても、「TDDサイクルの状態を構造化してファイルに持つ」設計は有効だ。セッションを跨ぐ場合。別の端末から作業を再開する場合。別の人間が途中から引き継ぐ場合。コンテキストウィンドウの外にある情報を扱う限り、外部記憶は必要になる。

## tokenの話

前回の記事で「Maxプランのx20でも、全部を毎回回したら3日で上限に達する」と書いた。

phase-compactはtoken消費の削減にも効いている。

7フェーズあるTDDサイクルで、コンテキストを圧縮せずに走ると、最後のcommitフェーズではコンテキストウィンドウの100%近くを使い切る。ここにreviewの並列エージェントが入ると、各エージェントが持つコンテキストも肥大化する。

正確な数字は計測していない。ただ、phase-compactを入れる前は中規模タスク（7-10ファイル変更）でコンテキスト溢れが発生していたのが、入れた後は発生しなくなった。体感で半分以下。

JetBrains Researchの研究（NeurIPS 2025）で面白い知見がある。コンテキスト管理の手法として、LLMに要約させるよりも、単純にツール出力を隠すだけ（Observation Masking）で同等の性能が出る、というものだ。要約のコストを払わなくても、必要な情報だけを残せばいい。phase-compactがやっていることと方向性は近い。フェーズごとに「何が要るか」を定義して、それ以外を捨てる。

## 忘れることを設計する

dev-crewを作っていて気づいたことがある。AIに何を覚えさせるかより、何を忘れさせるかの方が重要だ。

コンテキストウィンドウに入っている情報が多いほど、AIの判断精度は下がる。関係ない情報がノイズになるからだ。設計議論の詳細が残っていると、実装時にその議論を参照して「でも、こういうケースも考えた方が」と脱線する。

設計フェーズでは設計の情報だけ。実装フェーズでは実装に必要な情報だけ。レビューフェーズではレビューに必要な情報だけ。前のフェーズの詳細を意図的に消すことで、今のフェーズに集中させる。

「これを作れ。テストはここにある。設計の背景が必要ならここを読め」。phase-compactは、フェーズ遷移のたびにこの情報の選別をやっている。Phase Summaryの4行に収まらない情報は、次のフェーズでは不要だと判断している。

LLMのコンテキストウィンドウは有限だ。これは制約ではなく前提。前提だと思うと設計が変わる。「もっと大きなコンテキストウィンドウがほしい」ではなく、「この中で仕事を完結させるにはどうすればいいか」。答えは、仕事をフェーズに分けて、フェーズごとにコンテキストをリセットすること。

TDDサイクルはもともとフェーズが明確だ。RED、GREEN、REFACTORの境界がはっきりしている。この境界を「コンテキストの区切り」として使う。コンテキストウィンドウは制約じゃない。フェーズの区切りだ。
