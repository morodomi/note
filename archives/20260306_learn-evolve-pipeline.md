# LLMの自己改善にLLMは要らない。awkとgrepで十分だった

前回、セキュリティスキャンの話で「閾値0.95は経験で決めた」と書いた。誤検知フィルタの閾値。3週間、誤検知に溺れた経験から出た数字だ。

この「経験からルールを抽出する」部分を自動化した。LLMは使っていない。

---

## CLAUDE.mdのメンテが仕事になった

dev-crewのルールは全部CLAUDE.mdに書いている。「テストを先に書け」「セキュリティスキャンを忘れるな」「コミットメッセージはこの形式で」。

最初はうまくいった。でもプロジェクトが増えるたびにルールが増える。risk-classifier.shの閾値、false-positive-filterの0.95、phase-compactの保存項目。これらは全部手動で決めた。

ルールが増えると矛盾が出る。矛盾を直すためにまたルールを書く。設定ファイルのメンテナンスが仕事になっていた。

このシリーズで書いてきた3つの機能、phase-compact、リスクベースレビュー、セキュリティスキャン。どれも閾値やルールは人間が手で決めている。パフォーマンスチューニングのたびにCLAUDE.mdを開く。新しいプロジェクトで同じルールをまた書く。

人間が書き続けるのは持続しない。AIが自分の経験から学べばいい。

## 「変えた」操作だけを記録する

observe.shというPostToolUseフックを作った。Claude Codeのツール実行後に自動で走る。44行のbashスクリプト。

記録するのはファイル編集とシェルコマンドだけ。読み取り系は全て除外する。ls、cat、head、tail、grep、find、wc、echo、pwd、stat、readlinkなど28種の読み取り専用コマンドをgrepのパターンマッチで弾いている。

なぜここまで絞るのか。1セッションで数百回のファイル読み取りが走る。その大半はパターン抽出に使えない。「何かを変えた」操作だけが学びの材料になる。

記録フォーマットはJSONL。1行1イベント。timestamp、session_id、tool_name、targetの4フィールドだけ。

```json
{"timestamp":"2026-02-19T10:30:00Z","session_id":"abc-123","tool_name":"Edit","target":"src/index.ts"}
{"timestamp":"2026-02-19T10:31:00Z","session_id":"abc-123","tool_name":"Bash","target":"pnpm test"}
```

現在、18,000行を超えるログが溜まっている。設計で一番悩んだのは「何を記録するか」ではなく「何を捨てるか」だった。

## gitコマンド100回が高スコアになった日

ログからパターンを抽出するのがlearnスキルだ。3つの入力ソースを横断して探す。observeのツール使用ログ、gitのログとdiff、そしてCycle doc。

最初は回数だけでスコアリングしていた。「同じ操作が3回以上出現したらパターン」。

失敗した。gitコマンドが100回出現して最高スコアになった。誰でもやる操作だ。学んでも意味がない。

TF-IDFに着想を得て、独自のスコアリングを作った。セッションを文書、操作パターンを単語に見立てる。tfidf-summary.shという160行のawkスクリプトで計算する。

```bash
# TF = セッション内での出現割合の平均
tf = tf_sum / total_sessions

# IDF = log2(全セッション数 / (そのパターンが出現したセッション数 + 1))
idf = log(total_sessions / (term_sessions[t] + 1)) / log(2)

tfidf = tf * idf
```

全セッションに出現する操作はIDFが下がってスコアが減衰する。限られたセッションにだけ現れる操作はスコアが上がる。「稀だが再現性がある」パターンだけが残る設計だ。

ただしセッション数が20未満の初期段階ではTF-IDFが不安定になる。この場合は回数ベースのスコアリングにフォールバックする。データが溜まれば自動的に切り替わる。

## 14個の本能

見つかったパターンは「instinct（本能）」という構造化データに変換される。現在14個が蓄積されている。

いくつか紹介する。

**テストRED-GREENサイクル（confidence 0.7）**

テスト実行後に失敗を検出したら、ソースを修正して同じテストを再実行する。4つのセッションで繰り返し観測された。TDDの基本動作だが、AIが自分で「このパターンは重要だ」と判定した。

**Bashエラー出力マージ（confidence 0.7）**

コマンド実行時に標準エラー出力を標準出力にマージする。5つのセッションで観測。エラーが見えないまま次の処理に進む問題を防ぐパターンを、AIが自分で学習した。

**コミット前全テスト実行（confidence 0.7）**

git commitの前にtest-*.shをループで一括実行する。2つのセッションで観測。これもTDDの基本だが、CLAUDE.mdに書かなくてもAIが自分で抽出した。

**PHPStan実行（confidence 0.5）**

PHPファイルを複数回編集した後に静的解析を実行する。1セッションでしか観測されておらず、信頼度は低い。0.5は「まだ仮説段階」を意味する。

confidenceは2軸の掛け算で決まる。TF-IDFスコアからベーススコアを決め、観測回数で係数をかける。TF-IDFが0.2未満ならbaselineとして破棄。観測が増えるほど係数が上がり、31回以上で最大値になる。PHPStanの例のようにconfidence 0.5のinstinctは「まだ証拠が薄い仮説」として保存されるが、次のevolveでスキル化されることはない。0.8を超えて初めてMEMORY.mdへの昇格候補としてユーザーに提示する。

instinctとMEMORY.mdは二層構造だ。instinctは自動抽出された未検証の短期パターン。MEMORY.mdは人間が検証した長期知識。短期の仮説と確定した知識を分離している。

## 似たパターンをスキルに育てる

instinctが溜まると、似たパターンが目につくようになる。evolveスキルがこれをクラスタリングしてスキルにまとめる。

アルゴリズムはルールベースにした。

1つ目、domainフィールドで初期グルーピング。TypeScript、PHP、セキュリティなど、ドメインが同じinstinctを集める。

2つ目、triggerのキーワード重複率を計算。トークン化して共通トークンの割合を出す。50%以上一致なら「類似」と判定。

3つ目、類似ペアから連結成分を構成し、3件以上でクラスタ採用。

なぜLLMベースのクラスタリングにしなかったのか。instinctの数がそもそも14個だ。14個をクラスタリングするのにLLMのAPIコールを投げるのは、オーバーエンジニアリングの見本だ。domainの文字列比較とキーワードの重複率計算で十分。

クラスタが認定されると、スキル定義ファイルが自動生成される。元になったinstinctのIDが埋め込まれるので、「なぜこのスキルが生まれたのか」を後からトレースできる。

ここで重要なのが人間の承認だ。自動生成されたスキルが勝手に有効になるわけではない。ユーザーが確認し、承認して初めて取り込まれる。さらにテストゲートがある。全テストが通らなければロールバックされる。

## LLMを使わない部分の設計

このシリーズで4つの機能を書いてきた。振り返ると、全てに「LLMを使わずに済ませた部分」がある。

**phase-compact**: Cycle docはMarkdownファイル。フェーズごとの保存項目は事前定義。LLMが判断する余地をなくした。

**risk-classifier.sh**: 88行のbash。grepでファイルパスとdiffの内容を検索して点数をつける。変更のたびに判定結果が変わらない。

**false-positive-filter**: 誤検知の閾値0.95は数値1つ。テストコードかどうか、フレームワークの保護があるかどうか。判定ロジックは条件分岐。

そしてobserve.shとtfidf-summary.shに至っては、PostToolUseフックとawkスクリプトだけだ。ログの記録も、TF-IDFの計算も、LLMは1行も使っていない。

LLMにはLLMにしかできない仕事だけをやらせている。コードレビューで「このロジックは正しいか？」と問う。攻撃エージェントに「このコードをどう攻撃するか？」と問う。observerエージェントに「このログからどんなパターンが見えるか？」と自然言語で記述させる。

判定、計算、分岐、記録。これらはbashとawkとgrepでいい。LLMの出力は非決定的だ。同じ入力でも違う結果が返る。閾値の判定やリスクスコアの計算のような「毎回同じ結果であるべき処理」にLLMを使うのは設計ミスだ。

初回の記事で「2週間後にはまた半分くらい作り替えているかもしれない」と書いた。実際に作り替えた。11エージェントのレビューを2-5人に減らし、セキュリティスキャンに誤検知フィルタを足し、observe-learn-evolveの自己学習パイプラインを組み込んだ。

でも設計の原則は変わっていない。LLMには曖昧な判断を任せ、確定的な処理はシェルスクリプトで書く。この切り分けだけは、最初から今まで一貫している。

## シリーズを振り返って

dev-crewの価値はコードではない。コードは変わり続ける。変わらないのは設計思想だ。

- コンテキストウィンドウは制約ではなくフェーズの区切り
- レビューの深さはリスクに比例させる
- セキュリティは攻撃者の視点で検証する
- 自己改善はLLMではなく古典的な統計で十分

dev-crewはGitHubで公開している。ただ、コードは半年で腐る。設計思想は残る。dev-crewのコードではなく、「LLMに何をやらせないか」という切り分け方だけ持って帰ってもらえればいい。

- 第1回: コンテキストウィンドウは制約じゃない。フェーズの区切りだ

- 第3回: AIにセキュリティレビューさせるなら、防御側ではなく攻撃側をやらせろ
