# Claude Codeを手なずけた。暴走AIにTDDを仕込んだら別人になった話

タグ候補: #ClaudeCode #TDD #AI開発 #tdd-skills

Claude Codeは優秀だ。優秀すぎる。

バグを直してほしいだけなのに、周辺コードまでリファクタリングして、別の箇所を壊す。「テスト追加しておきました」と言いながら、頼んでいないテストフレームワークの移行まで始める。

欲しい修正を欲しい分だけ。それができない。

以前、この暴走について詳しく書いた。

- [AIの首輪はすぐ抜ける](https://note.com/morodomi/n/n20787aa82620)

禁止指示という首輪は効かなかった。だからレールを敷いた。SPECや仕様書、テスト。AIが走る方向を、仕組みで制限する。

この記事では、自分がClaude Codeで開発するときの実際のフローを見せる。

---

## なぜスコープを超えるのか

Claude Codeに「この関数を直して」と指示する。すると、その関数だけでなく、呼び出し元も、テストも、ドキュメントも直そうとする。

親切心ではない。スコープが見えていない。

「直して」だけでは、どこまでが修正範囲か分からない。だから全部やろうとする。コンテキストサイズには限界があるから、途中から部分最適化が始まる。不要な設計パターンが入り込む。頼んでいないリファクタリングが走る。

仕様書があれば、範囲が明確になる。テストがあれば、何を守るべきか分かる。

tdd-skillsは、このガイドをワークフローとして強制するスキル群だ。

```
INIT → PLAN → RED → GREEN → REFACTOR → REVIEW → COMMIT
```

各フェーズで「やること」と「やってはいけないこと」を定義している。INITで実装しない。REDではテストだけ書く。GREENでは最小限の実装だけ。AIはワークフローの上を走る。

---

## 自分の開発フロー

ここからは実際の会話ログを見せる。題材はtdd-skills自身の開発だ。メタだが、一番手元にあるログなので。

### 相談から始める

```
User:
ちょっと相談。
onboardは良くなったが、たまに古いスキルを参照して、
agent_docsを作ることがある。
version管理しないで、全て1.0.0またはバージョン無しで
リリースした方が良いのか？

reviewフェーズではquality-gateを基本的に通して欲しい。

commitフェーズでは、ドキュメントを更新してからコミットして欲しい。
先にコミットしてから、後で気がついて更新してコミットしている気がする。

これらの情報と、現在のスキル内容を参照して、相談にのってほしい。
```

「相談にのってほしい」と言う。「実装して」とは言わない。そうしないとAIは勝手に実装を始める。

Claudeは現状のファイルを読んで、課題を整理してくれる。

### issue化を指示する

やるべきことが見えたら、issue化を指示する。

```
User:
バージョンフィールドが不整合を起こすので、削除すること
quality-gateを必ず通すこと。
commitフェーズの順番更新
をissueに入れて
```

```
Claude:
3件のissueを作成しました:
- #37 - plugin.jsonからversionフィールドを削除
- #38 - tdd-reviewでquality-gateを必須実行にする
- #39 - tdd-commitでドキュメント更新をコミット前に移動
```

GitHub CLIが使える環境なら、Claudeがそのままissueを作ってくれる。

### TDDサイクルを回す

issueができたら、TDDサイクルを開始する。

```
User:
#37 から対応して tdd-init
```

tdd-initスキルが起動して、INITフェーズが始まる。プロジェクト状態を確認し、設計ドキュメントであるCycle docを生成する。INITでは実装しない。設計ドキュメントを作る。それ以上のことはしない。

### PLAN → RED → GREEN → REFACTOR → REVIEW → COMMIT

以降はスキルを順番に呼び出す。

```
User: tdd-plan
```

PLANフェーズでTest Listを作成する。何をテストするか、ここで決める。#37の場合は「plugin.jsonにversionフィールドが存在しないこと」「既存の機能が壊れないこと」あたりがテスト対象になる。

```
User: tdd-red
```

REDフェーズでテストを書く。この時点ではテストは失敗する。versionフィールドの削除がまだだから当然だ。

```
User: tdd-green
```

GREENフェーズで最小限の実装をする。テストが通るまで。ここでは「versionフィールドを消す」だけ。それ以上のことはしない。

```
User: tdd-refactor
```

テストが通った状態でコードを整理する。

```
User: tdd-review
```

quality-gateでカバレッジや静的解析の自動検証を実行し、基準を満たすか確認する。通らなければ差し戻される。

```
User: tdd-commit
```

ドキュメントを更新してからコミット。1チケット完了だ。

---

## 指示のコツ

短く指示する。 `tdd-red` だけで十分だ。「REDフェーズに入ってください。テストを書いてください」は長すぎる。スキルが何をすべきか知っている。余計な指示はむしろ混乱する。

相談と指示を分ける。「ちょっと相談」と言えば、Claudeは分析モードに入る。「tdd-init」と言えば、実行モードに入る。モードを明示する。

おかしいと思ったら止める。AIは止めないと止まらない。違和感があれば、すぐ介入だ。

---

## デメリット

正直、今でも面倒なときはある。

毎回スキルを呼び出す。サイクルを回す。typo直すだけなのに5分かかる。「手で書いたほうが早くない？」と何度も思った。

それでも使い続けている理由は単純だ。1回の暴走で半日潰れるより、毎回5分かけるほうがトータルで速い。

---

## 変わったこと

地味に嬉しいのは、テストが積み重なること。

以前はコードを書いてからテストを書いていた。開発優先のプロジェクトでは、テストはおざなりになる。どの企業でもありそうな話だ。

「テストファーストは理想だけど、現実には難しいよね」

そう思っていた。Claude Codeが出る前は、理想論だと思っていた。

AIにワークフローを渡したら、それが現実になった。

小さく進むので巻き戻しも楽だ。何かおかしくなっても、直前のテストまで戻れる。テストが資産として残る。副産物だったが、一番大きい変化かもしれない。

---

## まとめ

AIは猟犬だ。指示に従えば全力で走る。ただし、方向を示さなければ勝手に獲物を追いかける。

ワークフローを渡したら、ようやく一緒に狩りができるようになった。今では、チケットを投げると「まず仕様を確認します」と返ってくる。以前の「全部やっておきました」とは別人だ。頼んでいないリファクタリングも、テストフレームワークの移行も起きなくなった。

```
INIT → PLAN → RED → GREEN → REFACTOR → REVIEW → COMMIT
```

最初は面倒に感じるかもしれない。慣れてくると手放せなくなる。
