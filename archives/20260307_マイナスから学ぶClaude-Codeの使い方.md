# 壊して覚えるClaude Code入門

Claude Codeの正しい使い方を教える記事はたくさんある。この記事では逆をやる。まず壊す。そこから学ぶ。

お題はTODOアプリ。localhostで動くやつを作る。

前提として、ターミナルでcdやmkdirが使えること。Claude Codeのインストールがまだなら、公式の[セットアップガイド](https://code.claude.com/docs/ja/setup)を見てほしい。curlでインストールできる。

---

## 準備

空のディレクトリを作って、その中にCLAUDE.mdというファイルを1つだけ置く。

```
mkdir todo-app
cd todo-app
echo "日本語で返答すること" > CLAUDE.md
```

CLAUDE.mdはClaude Codeがプロジェクトのルールとして読み込むファイル。今は「日本語で返答すること」とだけ書いた。他のルールは一切書かない。

これだけで準備は終わり。

---

## ステップ1: 制約ゼロで作らせる

Claude Codeを起動して、こう投げる。

「TODOアプリを作って。localhostで動かしたい」

言語もフレームワークも指定しない。全部Claude Codeに任せる。

おそらくこうなる。

Claude Codeがフレームワークを勝手に選ぶ。ReactかNext.jsかVueか、そのときの気分で変わる。「TODOアプリ」としか言っていないのに、認証機能やダークモード切り替えを追加することもある。ファイルは20個以上生成される。テストは書かないことが多い。

なお、最近のClaude Codeはファイル作成やコマンド実行のたびに許可を求めてくる。全部手動で承認してもいいし、auto modeで全許可にしてもいい。ここでは制約ゼロの挙動を見たいので、許可が出たら全部通してほしい。

localhostで起動してみてほしい。動くはず。見た目もそれなりにきれいだと思う。

ここで「すごい、もう完成した」と思うかもしれない。実際、動くものが数分で出てくるのはすごい。でも、この状態には3つの問題が隠れている。それはステップ2で見えてくる。

---

## ステップ2: 機能を追加してみる

続けてこう頼む。

「期限機能を追加して」

追加自体はされる。期限の入力欄が出てきて、日付が表示されるようになる。

でも、ここで既存の機能を触ってみてほしい。TODOの追加、完了、削除。全部ちゃんと動くか？

環境やタイミングによるけど、既存機能が壊れることがある。完了ボタンが反応しなくなったり、削除したはずのTODOが残っていたり。私が試したときは、期限の入力欄を追加したら、既存のTODO一覧のレイアウトが巻き込まれて崩壊した。どれがどのTODOなのか判別できない。1つ機能を足しただけで、触っていない場所が壊れる。壊れなかったとしても、壊れていないことを確認する手段がない。目で見て手で触って確かめるしかない。これが地味にきつい。

もう1つ。「期限機能を追加して」と頼んだだけなのに、ソート機能や期限切れの色分けまで追加されていることがある。頼んでいないのに。

---

## ここまでで起きたこと

動くものはできた。でも3つの問題がある。

**何が入っているかわからない。** ファイルが20個以上もあって、どれが何をしているのか把握できない。フレームワークの選定理由もわからない。

**変更すると壊れるかもしれない。** テストがないから、変更のたびに手動で全機能を確認するしかない。確認漏れがあれば、壊れたまま気づかない。

**次に何を頼めばいいかわからない。** 設計がないから、Claude Codeに何を頼めばいいのか、自分でも判断できない。頼めば何か作ってくれるけど、それが正しい方向なのかわからない。

これが「暴走」の正体。AIが悪いわけじゃない。制約がないから起きる。「TODOアプリを作って」は、人間の同僚に言ったら「もうちょっと詳しく教えて」と返ってくる。制約を弱くしたClaude Codeは、聞き返さずに自分の判断で先に進む。それが便利でもあり、危険でもある。

制御する方法は3つある。

---

## ステップ3: CLAUDE.mdにルールを書く

最初の制御方法は、CLAUDE.mdにルールを追加すること。

さっきは「日本語で返答すること」しか書かなかった。ここにプロジェクトのルールを追加する。

例えばこう書く。

```
日本語で返答すること

## ルール
- 使用する技術はHTML、CSS、JavaScriptのみ。フレームワークは使わない
- 新しいファイルを作る前に、何を作るか説明して確認を取ること
- 機能追加は指示されたものだけ。勝手に追加しない
```

Claude Codeは毎回このファイルを読んでから作業を始める。ルールが書いてあれば、それに従う。

さっきのTODOアプリのディレクトリを一度消して、このCLAUDE.mdで最初からやり直してみてほしい。

「TODOアプリを作って。localhostで動かしたい」

今度はReactやVueは出てこない。HTML、CSS、JavaScriptだけで作る。ファイルを作る前に「index.htmlとstyle.cssとapp.jsの3ファイルを作ります」と確認してくるはず。認証もダークモードも追加されない。

同じ指示でも、CLAUDE.mdにルールがあるだけで結果が変わる。

ただし、CLAUDE.mdだけでは制御しきれないこともある。「何を作るか」の全体像がClaude Codeに伝わっていないと、個々のルールは守りつつも、全体としてちぐはぐなものができる。次のステップで、その問題を解決する。

---

## ステップ4: plan modeで設計してから作る

2つ目の制御方法は、実装の前に設計をすること。

Claude Codeにはplan modeがある。Shift+Tabを押すと入力モードがauto、plan、manualと切り替わる。planを選ぶと、ファイルの変更を行わず、設計だけを議論できる。

plan modeに入って、こう伝える。

「TODOアプリを作りたい。機能は追加、完了、削除の3つだけ。HTML、CSS、JavaScriptで作る」

Claude Codeが設計案を出す。ファイル構成、データの持ち方、画面のレイアウト。この段階ではコードは一切書かれない。

設計を見て、納得できたら承認する。納得できなければ修正を求める。「LocalStorageじゃなくてメモリ上で持つだけでいい」とか「CSSは最小限でいい、見た目は後で考える」とか。

承認したら、Shift+Tabでモードを戻す。Claude Codeは承認された設計に沿って実装を始める。

設計を先に決めることで、スコープが制限される。「追加、完了、削除の3つ」と合意しているから、認証やソート機能が勝手に生えてくることはない。

CLAUDE.mdが「やってはいけないこと」を定義するなら、plan modeは「やること」の合意。両方あると、暴走の余地がかなり狭まる。

---

## ステップ5: テストを先に書かせる

3つ目の制御方法。これが最も強力。

Claude Codeにこう頼む。

「テストを先に書いて。テストが通る最小限の実装をして」

テストを先に書く。そのテストを通すためだけに実装する。TDD（テスト駆動開発）だ。

例えば「TODOを追加できること」というテストを先に書く。テストを実行すると、まだ実装がないから失敗する。次に、そのテストだけを通す最小限のコードを書く。テストが通ったら、次のテストに進む。この繰り返し。

Claude Codeが出してくるテストはこんな感じになる。

```js
test('TODOを追加できる', () => {
  addTodo('牛乳を買う');
  expect(getTodos()).toContain('牛乳を買う');
});
```

このテストが通る最小限の実装だけを書く。テストがあると、さっきの3つの問題をかなり小さくできる。

テストを読めば何が入っているかわかる。「TODOを追加できる」「完了にできる」「削除できる」。これがそのまま仕様書になる。

期限機能を追加したあとにテストを実行すれば、既存が壊れていないか一瞬で確認できる。手動で全部触る必要がない。

新機能もまずテストで定義する。「期限を設定できること」「期限切れのTODOが上に表示されること」。テストを書く時点で、何を作るのかが決まる。実装は自動的にテストの範囲に収まる。

「テストを先に書いて」の一言が、Claude Codeに対する強力な制約になる。テスト先行にすると、実装のスコープをテストが縛る。テストが通る最小限を書く方向に誘導できるので、意図しない機能追加が起きにくくなる。

ただし「テストを先に書いて」だけでは、Claude Codeがテスト→実装の順序を守らないこともある。ステップ3のCLAUDE.mdに「テストを先に書く。テストが通ってから次に進む」と書いておくと、より確実になる。3つの制御方法は単独でも効くが、組み合わせるとさらに効く。

---

## まとめ

- **CLAUDE.md** で「勝手なことはするな」と釘を刺す
- **plan mode** で「まず何を作るか話そう」と合意を取る
- **TDD** で「壊れていないか」を常に監視する

面倒だと思うかもしれない。でもこの3つを省略して、あとからファイル50個のプロジェクトでどこに何があるかわからなくなったのを片付けるほうがよほど面倒だ。私は9ヶ月かけてそれを学んだ。

制御方法を先に教えるのが普通の順序だ。でも壊してみないと、なぜ制御が必要なのかわからない。

まず壊す。それから直す。そのほうが早い。
