# Claude Codeのルール管理が破綻した。AIに自分で学ばせた話

CLAUDE.mdにルールを書いた。「テストを先に書け」「セキュリティスキャンを忘れるな」「コミットメッセージはこの形式で」。最初はうまくいった。

でも、プロジェクトが増えるたびにルールが増える。ルールが増えると矛盾が出る。矛盾を直すためにまたルールを書く。気づいたら設定ファイルのメンテナンスが仕事になっていた。

人間が設定を書き続けるのは持続しない。AIが自分で学べばいい。そう考えて、observe、learn、evolveの3段階パイプラインを自作のClaude Codeプラグインに組み込んだ。

## 人間はどう学ぶか

人間は経験からパターンを見つける。「このエラーが出たら、だいたい原因はここ」「このライブラリを使うときは、この設定が必要」。最初は意識してやっている。繰り返すうちに体が覚える。それがスキルだ。

経験を観察して、パターンを抽出して、スキルにまとめる。この3ステップをAIに実装できないか。

## observe: 行動だけを記録する

Claude CodeにはPostToolUseフックがある。ツール実行後に指定した処理を走らせる仕組みで、これを使って書き込み系の操作をJSONLファイルに自動記録している。

```json
{"tool_name": "Bash", "target": "pnpm test", "session_id": "abc-123", "timestamp": "..."}
{"tool_name": "Edit", "target": "src/index.ts", "session_id": "abc-123", "timestamp": "..."}
```

設計で一番悩んだのは「何を捨てるか」だ。

記録対象はファイル編集（Edit/Write）とシェルコマンド（Bash）だけ。ファイルの読み取りや検索はフックの対象外にしてある。さらにBashの中でも、ls、cat、head、tail、grep、find、wc、echo、pwd、stat、readlinkなど25種の読み取り専用コマンドを除外リストに入れている。

なぜここまで絞るのか。読み取り系まで記録すると、ノイズが爆発する。1セッションで数百回のファイル読み取りが走るが、その大半はパターン抽出に使えない。意味のある行動、つまり「何かを変えた」操作だけを残す。

記録フォーマットはJSONL。1行1イベントで、timestamp、session_id、tool_name、targetの4フィールドだけ。セッションIDがあるので、後のlearn工程でセッション内の行動を時系列で追える。記録先はユーザーのホームディレクトリ配下。プロジェクトのGitリポジトリは汚さない。

## learn: パターンを見つける

ログが溜まったら、次はパターン抽出だ。サブエージェントが3種類の入力ソースを横断して探す。

1. **observeのツール使用ログ**（JSONL）
2. **gitのログとdiff**（何をどう変えたか）
3. **cycle docs**（開発サイクルの記録ドキュメント）

単一ソースでは見えないパターンが、複数ソースの突き合わせで浮かび上がる。observeログで「同じファイルを3回編集した」ことがわかり、gitのdiffで「毎回同じ種類の修正だった」と確認できる、という具合だ。

パターン検出には4つのルールがある。

- **繰り返しファイル編集**: 同じファイルへのEdit/Writeが3回以上。設定ファイルの試行錯誤が典型例
- **コマンドシーケンス反復**: 同一セッション内でRead→Edit→Bash(test)のようなツール列が3回以上繰り返される
- **ツール選好**: 特定ツールの使用頻度が突出している。GrepよりExploreを多用する傾向など
- **セッション横断パターン**: 異なるセッションで同じファイルが編集されている。毎回修正が必要なファイルがある

見つかったパターンは「instinct（本能）」という構造化データに変換される。

```json
{
  "id": "inst-20260225-001",
  "trigger": "TypeScriptプロジェクトの新規作成時",
  "action": "ESM設定でtsconfigを生成する",
  "confidence": 0.7,
  "domain": "typescript",
  "evidence": ["cycle-0.md", "cycle-0.md (別プロジェクト)"]
}
```

ここで重要なのがconfidence（信頼度）の設計だ。初期は「何回観察したか」だけでスコアを付けていた。でもこれだと、gitコマンドを100回叩いたパターンが高スコアになる。誰でもやる操作をパターンとして学んでも意味がない。

現在は「珍しさ」と「回数」の掛け算でスコアリングしている。珍しさの計算にはTF-IDFの考え方を借りた。セッションを文書、操作パターンを単語に見立てる。全セッションに出現する操作はスコアが下がり、限られたセッションにだけ現れる操作はスコアが上がる。「稀だが再現性がある」パターンほど学ぶ価値がある、という設計だ。

回数が3回未満なら証拠不足で破棄。ありきたりな操作もTF-IDFが弾く。両方のフィルタを通過して初めてinstinctとして保存される。0.8を超えたものは、MEMORY.mdへの昇格候補としてユーザーに提示する。

なおセッション数が20未満の初期段階では、統計が不安定なため回数ベースのスコアリングにフォールバックする。データが溜まれば自動的にTF-IDFに切り替わる。

instinctとMEMORY.mdは二層構造になっている。instinctは自動抽出された未検証の短期パターン。MEMORY.mdは人間が検証した長期知識。短期の仮説と確定した知識を分離している。人間の「なんとなくそうかも」と「確実にこうだ」の違いを、そのまま設計に落とした。

## evolve: スキルに育てる

instinctが溜まってくると、似たパターンが目につくようになる。これをクラスタリングでスキルにまとめる。

アルゴリズムはルールベースにした。

1. **domainフィールドで初期グルーピング**。TypeScript、PHP、セキュリティなど、ドメインが同じinstinctを集める
2. **triggerのキーワード重複率を計算**。トークン化して共通トークンの割合を出す。50%以上一致なら「類似」と判定
3. **類似ペアから連結成分を構成し、3件以上でクラスタ採用**

なぜ機械学習ベースではなくルールベースなのか。instinctの数がそもそも少ない初期段階では、LLMを使ったクラスタリングはコストに見合わない。ルールベースで十分な段階でオーバーエンジニアリングはしない。

クラスタが認定されると、スキル定義ファイルが自動生成される。元になったinstinctのIDが埋め込まれるので、「なぜこのスキルが生まれたのか」を後からトレースできる。

ここで重要なのが、人間の承認ステップだ。自動生成されたスキルが勝手に有効になるわけではない。ユーザーが確認し、承認して初めてプラグインに取り込まれる。さらにテストゲートがある。全テストが通らなければロールバックされる。AIが勝手に進化するのではなく、人間のチェックポイントを挟んでいる。

## 自動で回る仕組み

学習を手動で実行するのでは意味がない。環境変数で有効化すると、コミット後に自動で回る。

条件は2つ。observeのログが20イベント以上記録されていること。そして前回の学習から一定時間が経過していること。条件を満たしたら、コミット完了後にlearnが自動実行される。デフォルトでは無効にしてある。意図せず学習が走るのを防ぐためだ。

失敗してもコミット自体はブロックしない。学習はベストエフォートだ。

## リスクと限界

誤学習。間違ったパターンを学んだらどうするか。三重のフィルタで軽減している。まず3回未満の観察は証拠不足として破棄。次にTF-IDFで「誰でもやる操作」は自動的にスコアが減衰する。そしてスキル化には人間の承認が必要だ。回数、特徴度、人間の目。この三段階で誤学習の混入を抑えている。

過学習。TypeScriptで学んだパターンがPythonで邪魔になるかもしれない。domainフィールドで分離しているが、ドメインをまたぐパターンへの対処はまだ弱い。

コスト。観察、学習、進化、それぞれにAPIコールが発生する。20件の閾値、デフォルト無効、ベストエフォート実行。3つの仕組みでオーバーヘッドを抑えている。

どのリスクも「解決はしていないが、軽減の仕組みは入っている」という状態だ。
