# AIが書くテストは動く。でも「仕様」として読めるか？

AIにテストを書かせると、量は出る。

私のプロジェクトは最初から最後までAIに作らせた。テストも全部AIが書いた。942テスト関数。個別に見れば、ちゃんと動く。カバレッジも悪くない。CI は全部緑。

でもある日、仕様を変更しようとして手が止まった。どのテストがどの仕様に対応しているのか、構造的に分からない。テスト名を見ても振る舞いが読めない。mockだらけで、実装を少し変えただけで関係ないテストが壊れる。

テストは通る。でも仕様書としては読めない。

この問題は、カバレッジでは見えない。カバレッジが100%でも、テストが実装の詳細に結合していたら仕様変更で全部壊れる。mutation testingでも見えない。テストの欠陥検出力は高くても、「このテストは仕様として読めるか？」とは別の話だ。

テストが「動くか」でも「漏れがないか」でもない、第三の軸がある。テストが「仕様として機能するか」だ。

---

## 既存の指標では足りなかった

整理するとこうなる。

- カバレッジ: コードがテストで触られているか。仕様との対応は見ない
- mutation testing: テストが欠陥を検出できるか。正確だけど遅い。仕様としての可読性は見ない
- コードレビュー: 人間が見れば分かる。でもAIが書く量に追いつかない

どれも大事だけど、「テストが仕様として読めるか」を測る道具がなかった。

以前[「テストの理想形を言語化したら、AIへの指示が変わった」](https://note.com/morodomi/n/nf9e820194410)という記事を書いた。あの記事で言語化した「テスト設計の4つの性質」は、言葉としては整理できた。ただ、言葉だけでは、AIが書いた942個のテストを1つずつ確認する問題は解決しない。

だからツールを作った。exspecという。テスト設計品質を静的に検証する。LLMは使わない。APIコストゼロ。CIに入れてミリ秒で回る。

---

## 何を見ているのか

exspecはテストコードを読んで、4つの性質に照らして指摘を出す。

- **What not How**: テストが振る舞いを記述しているか。実装の詳細に依存していないか
- **Living Documentation**: テストを読めば仕様が分かるか。別途ドキュメントが要らないか
- **Compositional**: 1つのテストが1つの責務を検証しているか
- **Single Source of Truth**: 同じ仕様が1箇所だけに書かれているか

正確に言うと、exspecが見ているのは「仕様として読みにくくなる構造的な兆候」だ。仕様としての可読性そのものを直接測っているわけではない。アサーションがないテスト、mockが過剰なテスト、巨大すぎるテスト。こういう構造的な臭いが、可読性を損なう。

4つの性質それぞれに対して検出ルールを設けた。合計16ルール、重要度で2つのTierに分類している。Tier 1は構造的な問題（アサーションなし、mock過多、巨大テスト）。Tier 2は設計上の臭い（実装結合、fixture肥大、曖昧な命名）。

Rustで書いた。Python、TypeScript、PHP、Rustに対応している。

## 使い方

自分のプロジェクトで走らせたときの出力がこれだ。

```
exspec v0.1.0 -- 71 test files, 942 test functions
WARN ./src/__tests__/api-client-retry.test.ts:11 T101 how-not-what
INFO ./src/__tests__/timeline-api.test.ts:45 T003 giant-test
Score: BLOCK 0 | WARN 12 | INFO 23 | PASS 907
```

BLOCKは優先的に見るべき強いシグナル。WARNはコンテキスト次第。INFOは参考程度。

インストールはcargoから。言語指定もできる。

```bash
cargo install --git https://github.com/morodomi/exspec.git
exspec .
exspec --lang typescript .
```

初めて使うなら、まずTier 1だけ有効にして、BLOCKを潰すところから始めるのがいい。Tier 2は設定ファイルで無効にできる。

```bash
exspec init --lang typescript  # .exspec.tomlを生成
```

## 自分のプロジェクトで走らせた結果

2つのプロジェクトで試した。どちらもAIが書いたテストだ。

**プロジェクトA（TypeScript、SaaS）**: 942テスト関数。BLOCKゼロ。WARNとINFOがいくつか出た。指摘をコードを書いたClaude Codeに見せたら、「これは直す価値がある」と認めたものもあった。

**プロジェクトB（Python、データ分析）**: 2208テスト関数。BLOCKが7件、WARNが229件。規模が大きい分、指摘も多い。

プロジェクトBのBLOCK 7件を手動で検証した。結果はTP（真の問題）4件、FP（誤検知）3件。TPの4件は全部「例外が出ないことを確認するだけのテスト」で、assertが一行もない。セットアップ確認のつもりで書いたのだろうが、仕様としては何も言っていない。FPの3件はpytest fixtureをテスト関数と誤認したものと、pytest.fail()を検証手段として認識できなかったもの。これはexspec側のバグだ。

2つのプロジェクトに共通して多かったWARN:

- T101 how-not-what -- mockの呼び出し検証が多すぎる。実装をリファクタしたらテストが壊れるタイプ
- T003 giant-test -- 50行超のテスト。プロジェクトBでは117件もあった
- T002 mock-overuse -- mock依存が多すぎる

的外れだった指摘:

- T109 undescriptive-test-name -- 「ローディング中にスケルトンが表示される」を曖昧な命名と判定した。英語のtest_xxx形式を期待しているだけで、日本語のテスト名は十分に振る舞いを記述している。これは誤検知
- T004/T005/T008 -- parameterized tests、PBT、contract testingがないという指摘。全テストファイルに求めるのはノイズ

面白かったのは、exspecの指摘をAIに見せると、AI自身が「自分のテストの弱点」を認識すること。たとえばT101 how-not-whatの指摘を見せたら、「mockの呼び出し順序に依存していて、振る舞いの検証になっていない。リファクタすべき」と自分で認めた。人間がレビューするより速い。v0.1.0なので誤検知もあるが、妥当な指摘も確実にある。

## 9プロジェクトでdogfoodingした

公開前に9つのOSSプロジェクト、約23,000テストで検証した。Python、TypeScript、PHP、Rust。

結果は言語とプロジェクトによってかなり違う。NestJSは誤検知0%だった一方、Pydanticは55%が誤検知。Rustはマクロの中のテストが見えないという根本的な制約がある。

全部の結果はGitHubのdocs/dogfooding-results.mdに載せてある。誤検知率も含めて全部公開している。

## まだv0.1.0

正直に言うと、まだ荒い。日本語テスト名の誤検知、Rustマクロの検出限界、ファイルレベルINFOのノイズ。全部分かっている。

AIがテストを量産する時代に、動くかどうかだけでは足りない。仕様として読めるかどうかを機械的にチェックする手段が要る。

CIに入れて回し続ける。精度は使いながら上げる。
