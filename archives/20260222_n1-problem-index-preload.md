# N+1問題はORMの話だと思っていた。166クエリを5に減らすまでの記録

N+1問題。Djangoならselect_related、Railsならincludes。ORM側の話だと思っていました。

半分だけ正解です。

N+1問題とは、ループの中でデータを1件ずつ取りに行くパターンです。10件なら10回、100件なら100回DBに問い合わせる。データが増えるほどクエリ数が膨れる。AIにコードレビューを任せていると、この指摘が出てくることがあります。厄介なのは、コードを生成する側のAIもこのパターンを書きやすいことです。

私はデータ分析パイプラインを運用しています。ORMは使っていません。生SQLでデータを取得して、複数の集計モジュールに渡して特徴量を算出する。フレームワークもない、素のPython + SQLiteの構成です。

そこにN+1問題が潜んでいました。

テーブル構造はこうなっています。

```
records  ── 1件の処理対象。日付で検索する
  └─ entries  ── recordsに紐づく明細(エントリ)。1レコードに15件程度
       ├─ entity_id   → 明細の主体情報
       └─ related_id  → 明細の関連情報
results  ── recordsに紐づく実績データ
```

---
## レビューで突きつけられた166という数字
自分では気づかなかった。

コードレビューを自動化しているのですが、そのレビューエージェントが指摘してきたんです。「インデックスが1本もない。FK列にすらない」と。

調べてみたら、1レコードを処理するたびにDBを166回叩いていた。

内訳はこうです。

```
1レコードの処理で何が起きているか:

  for entry in entries:        # 15件
    モジュールA(entry)  → 3クエリ
    モジュールB(entry)  → 2クエリ
    モジュールC(entry)  → 2クエリ
    モジュールD(entry)  → 2クエリ
    モジュールE(entry)  → 2クエリ
                          -------
    1エントリあたり       11クエリ

  11クエリ × 15エントリ = 165
  + レコード情報の取得     1
  合計                   166クエリ
```

1,000レコード処理すれば166,000クエリ。開発中は数十件で回していたから気づかなかった。データ量が増えた瞬間に破綻する構造です。

## まずインデックスだけ追加した
インデックス追加とクエリ統合、両方を一度にやりたかった。でも、やめた。

理由はリスクの分離です。インデックスの追加はコード変更がゼロ。テーブル定義にCREATE INDEX IF NOT EXISTSを5本足すだけ。壊れようがない。

```sql
CREATE INDEX IF NOT EXISTS idx_entries_record_id ON entries(record_id);
CREATE INDEX IF NOT EXISTS idx_entries_entity_id ON entries(entity_id);
CREATE INDEX IF NOT EXISTS idx_entries_related_id ON entries(related_id);
CREATE INDEX IF NOT EXISTS idx_records_date ON records(date);
CREATE INDEX IF NOT EXISTS idx_results_record_id ON results(record_id);
```

外部キー（FK）列と日付範囲検索にインデックスを張る。SQLiteはPRIMARY KEYとUNIQUE制約以外には自動でインデックスを作らないので、FK列が丸裸だったこと自体が問題でした。

テストは3本。インデックスが作られること。2回実行しても冪等であること。EXPLAIN QUERY PLANでインデックスが使われること。TDDサイクルで30分もかかっていません。

で、結果はどうだったか。

正直、体感は変わらなかった。

インデックスは既存クエリの実行を速くするだけです。166回叩いている事実は変わらない。個々のクエリが速くなっても、回数が減らないと根本的な解決にはならない。

## 次にクエリの回数を減らした
インデックスで個々のクエリは速くなった。ただ、166回という回数は変わっていない。1つのサイクルで2つの変更を同時にやりたくなかったから、インデックスが安全に動くことを確認してから次の手を打つ。

ここでやったのが一括プリロードです。

166クエリの構造を思い出してください。5つの集計モジュールが、エントリごとにバラバラにDBへアクセスしている。同じテーブルに何度も問い合わせている。

なら、先にまとめて取得すればいい。

1レコードの処理を始める前に、5回のクエリでそのレコードに必要なデータを全部メモリに載せる。レコード情報、全エントリ、関連エンティティ情報、過去の実績、集計済みの統計。

```
最適化前:
  for entry in entries:        # 15件
    SELECT ... WHERE id = ?    # 毎回DBを叩く（x5モジュール）
  → 166クエリ

最適化後:
  entries = SELECT ... WHERE record_id = ?    # まとめて1回
  entities = SELECT ... WHERE id IN (...)     # まとめて1回
  related = SELECT ... WHERE id IN (...)      # まとめて1回
  results = SELECT ... WHERE record_id = ?    # まとめて1回
  stats = SELECT ... WHERE record_id = ?      # まとめて1回
  → 5クエリ

  for entry in entries:
    モジュールA(entry, メモリ上のデータ)  # DBに触らない
```

各集計モジュールは、メモリ上のデータを参照する。DBには触らない。データがなければ従来通りDBから取得するので、後方互換も保てる。

最適化前: 1レコード(15エントリ)あたり166クエリ。1,000レコードで166,000クエリ。
最適化後: 1レコードあたり5クエリ。1,000レコードで5,000クエリ。削減率97%。

166が5になった。

ただし、こちらは簡単ではなかった。5つの集計モジュールを全部修正する必要がある。1モジュール変えるだけなら楽ですが、5つ同時に触るとリグレッションのリスクが跳ね上がる。

テストは7本書きました。正常系4本、境界値2本、異常系1本。プリロード結果が正しいか、最適化前と差分がないか、クエリ数が5以下に収まっているか。この3点を検証する構成です。

## なぜ2回に分けたのか
「最初から一括プリロードだけやればよかったのでは」と思うかもしれません。

設計レビューの段階で、AIが2つ同時にやろうとしたんです。インデックスもプリロードも一気にやる計画を出してきた。

レビューで弾きました。スコープが大きすぎる。5つのモジュールを同時に修正する変更と、純粋にadditiveなインデックス追加を混ぜるな、と。

結果的にこの判断は正しかった。インデックスのサイクルは30分。プリロードのサイクルは、テストの設計だけで1時間以上かかっている。リスクの大きさが全然違う。

小さい変更を先に安全に入れて、次の変更の足場にする。TDDでサイクルを分ける判断は、技術的な正しさだけじゃなく、リスクのコントロールでもあります。

## 「ORMを使えばいい」への回答
「Djangoならselect_related一発だろ」。はい、その通りです。ORMを使っているなら。

世の中にはORMを使わないコードがある。データ分析パイプライン、バッチ処理、レガシーシステム。生SQLを書いている現場は思っている以上に多い。

そういう環境でN+1が起きたとき、「ORM導入しましょう」は現実的な解じゃない。既存のコードを壊さずに、インデックスとプリロードで対処する。やったのはそれだけです。

「特殊なケースだろ」という反論もあると思います。SQLite + 生SQLの構成は確かにニッチです。ただ、N+1の正体は「ループの中でクエリを投げている」こと。それはORM経由でもSQL直書きでも変わらない。

## 学んだこと
166クエリを5にした話でしたが、大事だったのは2つです。

レビューで問題に気づけたこと。そして、修正を2段階に分けてリスクをコントロールしたこと。

97%という数字より、そっちのほうが再現性がある。ORMの有無に関係なく。
